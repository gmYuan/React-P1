# React Hooks 实现与内部数据共享层完整指南

## 第一部分：顶层鸟瞰

### 背景问题

在 React 中实现 Hooks 功能时，面临以下核心挑战：

**核心问题：**
- Hook 脱离 FC（Function Component）上下文，作为普通函数如何拥有感知上下文环境的能力？
- Hook 如何知道在另一个 Hook 的上下文环境内执行？
- Hook 如何知道当前是 mount 还是 update 阶段？
- Hook 如何知道自身数据保存在哪里？

### 宏观图

**Hooks 数据流架构：**

```
Reconciler                内部数据共享层                React
┌─────────────┐          ┌────────────────┐          ┌────────┐
│  mount时    │          │                │          │        │
│             │─────────>│  当前使用的    │─────────>│ React  │
│  useState   │          │  Hooks集合     │          │        │
│  useEffect  │          │                │          │        │
│     ...     │          └────────────────┘          └────────┘
│             │
│  update时   │
│             │
│  useState   │
│  useEffect  │
│     ...     │
│             │
│ hook上下文中│
│             │
│  useState   │
│  useEffect  │
│     ...     │
└─────────────┘
```

**Hook 数据存储结构：**

```
FC FiberNode
     │
     ├─ memoizedState ──> useState ──> hook数据
     │                       │
     │                       ↓
     │                   useEffect
     │                       │
     │                       ↓
     │                   useState
     │
     └─ updateQueue
```

**对于 FC 对应的 fiberNode，存在两层数据：**
1. `fiberNode.memoizedState` 对应 Hooks 链表
2. 链表中每个 hook 对应自身的数据

### 关键技术栈

| 技术 | 说明 |
|------|------|
| **内部数据共享层** | Reconciler 与 React 的桥梁，使不同上下文的 Hook 能共享 Dispatcher |
| **Dispatcher** | Hook 集合接口，定义 useState 等方法，在不同阶段切换不同实现 |
| **currentDispatcher** | 当前使用的 Dispatcher 实例，通过内部数据共享层解析获得 |
| **currentlyRenderingFiber** | 记录当前正在 render 的 FC 对应 fiberNode |
| **workInProgressHook** | 记录当前正在处理的 Hook |
| **HooksDispatcherOnMount** | mount 阶段的 Dispatcher 实现 |

### 步骤 1：创建 currentDispatcher.ts 文件

**路径：** `packages/react/src/currentDispatcher.ts`

**修改内容：**
1. 导入 Action 类型
2. 定义 Dispatcher 接口，包含 useState 方法
3. 定义 Dispatch 类型
4. 创建 currentDispatcher 对象用于存储当前 Dispatcher
5. 实现 resolveDispatcher 函数，用于解析并获取当前 Dispatcher

```typescript
import { Action } from 'shared/ReactTypes';

export interface Dispatcher {
  useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>];
}

export type Dispatch<State> = (action: Action<State>) => void;

const currentDispatcher: { current: Dispatcher | null } = {
  current: null
};

export const resolveDispatcher = (): Dispatcher => {
  const dispatcher = currentDispatcher.current;

  if (dispatcher === null) {
    throw new Error('hook只能在函数组件中执行');
  }
  return dispatcher;
};

export default currentDispatcher;
```

## 第三部分：主要知识点

### 1. Hook 如何感知上下文环境

**问题：** hook 脱离 FC 上下文，仅仅是普通函数，如何让它拥有感知上下文环境的能力？

比如说：
- hook 如何知道在另一个 hook 的上下文环境内执行？

```javascript
function App() {
  useEffect(() => {
    // 执行 useState 为什么能返回正确的值
    useState();
  });
}
```

- hook 怎么知道当前是 mount 还是 update？

**解决方案：** 在不同上下文中调用的 hook 不是同一个函数数。

在不同的上下文中，Reconciler 会根据当前阶段（mount/update）或所处位置（hook上下文中）注入不同的 Dispatcher 实现。

```
Reconciler                内部数据共享层
┌─────────────┐          ┌────────────────┐
│  mount时    │          │                │
│             │          │  当前使用的    │
│  useState   │─────────>│  Hooks集合     │─────────> React
│  useEffect  │          │                │
│     ...     │          └────────────────┘
│             │
│  update时   │
│             │
│  useState   │
│  useEffect  │
│     ...     │
│             │
│ hook上下文中│
│             │
│  useState   │
│  useEffect  │
│     ...     │
└─────────────┘
```

### 2. 内部数据共享层的作用

**实现「内部数据共享层」的注意事项：**

**目的：** 以浏览器举例，Reconciler + hostConfig = ReactDOM

增加「内部数据共享层」，意味着 Reconciler 与 React 产生关联，进而意味着 ReactDOM 与 React 产生关联。

如果两个包「产生关联」，在打包时需要考虑：两者的代码是打包在一起还是分开？

**如果打包在一起，** 意味着打包后的 ReactDOM 中会包含 React 的代码，那么 ReactDOM 中会包含一个内部数据共享层，React 中也会包含一个内部数据共享层，这两者不是同一个内部数据共享层。

**而我们希望两者共享数据，** 所以不希望 ReactDOM 中会包含 React 的代码。

### 3. hook 如何知道自身数据保存在哪里？

**答案：** 可以记录当前正在 render 的 FC 对应 fiberNode，在 fiberNode 中保存 hook 数据。

```javascript
function App() {
  // 执行 useState 为什么能返回正确的值
  const [num] = useState(0);
}
```

**实现 Hooks 的数据结构**

fiberNode 中可用的字段：
- **memoizedState**
- **updateQueue**

对于 FC 对应的 fiberNode，存在两层数据：
1. `fiberNode.memoizedState` 对应 Hooks 链表
2. 链表中每个 hook 对应自身的数据

**数据结构示意：**

```
FC FiberNode
     │
     ├─ memoizedState ──> useState ──> hook数据
     │                       │
     │                       ↓
     │                   useEffect
     │                       │
     │                       ↓
     │                   useState
     │
     └─ updateQueue
```

### 4. useState 实现原理

**useState 的调用流程：**

1. **在 React 包中定义 useState：**
   ```typescript
   export const useState: Dispatcher['useState'] = (initialState) => {
     const dispatcher = resolveDispatcher();
     return dispatcher.useState(initialState);
   };
   ```

2. **通过 resolveDispatcher 获取当前 dispatcher：**
   ```typescript
   export const resolveDispatcher = (): Dispatcher => {
     const dispatcher = currentDispatcher.current;
     if (dispatcher === null) {
       throw new Error('hook只能在函数组件中执行');
     }
     return dispatcher;
   };
   ```

3. **在 renderWithHooks 中根据阶段设置 dispatcher：**
   ```typescript
   export function renderWithHooks(wip: FiberNode) {
     currentlyRenderingFiber = wip;
     wip.memoizedState = null;
     
     const current = wip.alternate;
     
     if (current !== null) {
       // update 阶段
     } else {
       // mount 阶段
       currentDispatcher.current = HooksDispatcherOnMount;
     }
     
     const Component = wip.type;
     const props = wip.pendingProps;
     const children = Component(props);
     
     currentlyRenderingFiber = null;
     return children;
   }
   ```

4. **mount 阶段的 useState 实现（mountState）：**
   需要实现：
   - 创建 hook 对象
   - 将 hook 添加到 hooks 链表
   - 初始化 hook.memoizedState
   - 返回 [state, dispatch] 元组

## 第四部分：个人 Q&A

### Q1: 为什么要创建 shared 包的 internals.ts 文件？它有什么作用？

**用最简单的话说：这是一座"桥"，连接 React 包和 Reconciler 包。**

#### 1.1 首先理解问题的本质

假设你要完成一个任务，分三步：

```
步骤1（React包）     →     步骤2（Reconciler包）     →     步骤3（React包）
准备工具箱                    使用工具箱干活                回收工具箱
```

**问题来了：** 步骤2（Reconciler）怎么拿到步骤1（React）准备的工具箱？

两个包是独立的，不在一起，怎么办？

#### 1.2 解决方案：internals.ts 就是"传递工具箱"的人

```typescript
// packages/shared/internals.ts
import * as React from 'react';

const internals = React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED;

export default internals;
```

**这段代码做了什么？**

1. **从 React 包导入整个 React 对象**
   ```typescript
   import * as React from 'react';
   ```
   就像说："我去 React 那里拿东西"

2. **获取 React 内部的"工具箱"**
   ```typescript
   const internals = React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED;
   ```
   这个超长的名字是 React 故意起的，意思是"这是内部 API，普通用户别用"
   
   这个 `internals` 对象里装着什么？就是 `currentDispatcher`！

3. **把工具箱导出，给其他包用**
   ```typescript
   export default internals;
   ```

#### 1.3 完整的流程图

```
React 包 (packages/react/index.ts)
   │
   │ 创建 currentDispatcher
   │ 放入 __SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED
   │
   ↓
Shared 包 (packages/shared/internals.ts)
   │
   │ 从 React 拿到 internals
   │ 里面包含 currentDispatcher
   │
   ↓
Reconciler 包 (packages/react-reconciler/src/fiberHooks.ts)
   │
   │ 从 shared/internals 获取 currentDispatcher
   │ 可以读取和修改它
   │
   └─→ 设置 currentDispatcher.current = HooksDispatcherOnMount
```

#### 1.4 如果没有 internals.ts 会怎样？

**方案A：直接导入内部模块** ❌
```typescript
// 在 fiberHooks.ts 中
import currentDispatcher from 'react/src/currentDispatcher';  // ❌ 不行！
```

为什么不行？因为打包时会把 `currentDispatcher` 模块的**源代码**复制一份到 Reconciler 包里：
- React 包打包后有一个 `currentDispatcher` 对象（在内存地址A）
- Reconciler 包打包后也有一个 `currentDispatcher` 对象（在内存地址B）
- **它们是两个不同的对象！** 互相看不到对方的修改

**方案B：导入已打包的 React** ✅
```typescript
// 在 internals.ts 中
import * as React from 'react';  // ✅ 导入的是打包后的 React
const internals = React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED;
```

为什么可以？因为：
- `import * as React from 'react'` 导入的是**已经打包好的 React 包**
- 不是导入源代码，而是导入**运行时的 React 对象**
- 这个 React 对象里已经包含了创建好的 `currentDispatcher`
- **所有人拿到的都是同一个运行时对象**

#### 1.4.1 深入理解：为什么两种导入方式结果不同？

**核心区别：导入的是"源代码"还是"打包后的包"**

```
方案A：导入内部模块（源代码）
===============================
打包过程：

React 包打包：
  src/currentDispatcher.ts  →  打包  →  dist/react.js
                                        └─ 包含 currentDispatcher 对象 (地址A)

Reconciler 包打包：
  src/fiberHooks.ts
  ├─ import from 'react/src/currentDispatcher'
  └─ 打包工具看到这个导入
     └─ 把 currentDispatcher.ts 的源代码复制进来
        └─ 重新执行一遍代码
           └─ 创建新的 currentDispatcher 对象 (地址B) ❌

结果：两个不同的对象！
```

```
方案B：通过 internals 导入（打包后的包）
=======================================
打包过程：

React 包打包：
  src/currentDispatcher.ts  →  打包  →  dist/react.js
  src/index.ts                         └─ 包含 currentDispatcher 对象 (地址A)
  └─ export __SECRET_INTERNALS_...       └─ 对外暴露这个对象

Reconciler 包打包：
  src/fiberHooks.ts
  ├─ import internals from 'shared/internals'
  └─ 在 internals.ts 中：
     └─ import * as React from 'react'  ← 引用已打包的 react.js
        └─ 获取 React.__SECRET_INTERNALS_...
           └─ 拿到的就是地址A的对象 ✅

结果：同一个对象！
```

#### 1.4.2 用配置文件举例

还记得步骤4中的配置吗？

```javascript
// scripts/rollup/react-dom.config.js
external: [...Object.keys(peerDependencies)]
```

这行配置告诉打包工具：
- **`external`：** "这些包不要打包进来，保持外部引用"
- **`peerDependencies` 包含 `react`**
- 所以打包时遇到 `import from 'react'`，不会复制代码，而是保持引用

**如果导入内部模块：**
```typescript
import currentDispatcher from 'react/src/currentDispatcher';
```
- 这**不是**从 `'react'` 包导入
- 是从 `'react/src/...'` 路径导入
- 打包工具不认为这是 external 包
- **会把源代码复制进来**

**如果通过 internals 导入：**
```typescript
import * as React from 'react';  // 在 internals.ts 中
```
- 这**是**从 `'react'` 包导入
- 匹配 external 配置
- 打包工具保持外部引用
- **不复制代码，运行时引用同一个对象**

#### 1.5 类比理解：菜谱 vs 做好的菜

**方案A：导入源代码（`import from 'react/src/...'`）**
- 就像你找朋友要**菜谱**
- 你按照菜谱自己做了一份菜（地址A）
- 朋友按照菜谱也做了一份菜（地址B）
- **两份菜是不同的**，你吃掉自己的菜，朋友的菜不会少

**方案B：导入打包后的包（`import from 'react'`）**
- 就像你和朋友**共享一盘菜**
- 朋友做好了菜放在桌上（React 包导出）
- 你通过 internals 拿到这盘菜的引用
- **是同一盘菜**，你吃了一口，朋友那边也会看到少了

```
方案A（导入源代码）：
┌─────────────┐         ┌─────────────┐
│  React 包   │         │ Reconciler  │
│             │         │    包       │
│ 菜谱 📄     │         │  菜谱 📄    │
│  ↓          │         │   ↓         │
│ 做菜 👨‍🍳    │         │  做菜 👨‍🍳   │
│  ↓          │         │   ↓         │
│ 红烧肉🍖(A) │  ❌     │ 红烧肉🍖(B) │
│             │         │             │
│ (不同的对象) │         │ (不同的对象) │
└─────────────┘         └─────────────┘

方案B（导入打包后的包）：
┌─────────────┐         ┌─────────────┐
│  React 包   │         │ Reconciler  │
│             │         │    包       │
│ 做好的菜    │         │             │
│ 红烧肉🍖────┼────────→│  拿到引用   │
│    (地址A)  │  ✅     │  (地址A)    │
│             │         │             │
│ (同一个对象) │         │ (同一个对象) │
└─────────────┘         └─────────────┘
```

#### 1.6 实际运行时的例子

假设运行时的情况：

**方案A（错误的）：**
```javascript
// React 包创建：
const currentDispatcher = { current: null };  // 内存地址: 0x001
React.__SECRET_INTERNALS = { currentDispatcher };

// Reconciler 包因为复制了源代码，重新创建：
const currentDispatcher = { current: null };  // 内存地址: 0x002 (新的！)

// React 包设置：
React.__SECRET_INTERNALS.currentDispatcher.current = mountDispatcher;
// 修改的是地址 0x001

// Reconciler 包读取：
console.log(currentDispatcher.current);  
// 读取的是地址 0x002，还是 null ❌
```

**方案B（正确的）：**
```javascript
// React 包创建并导出：
const currentDispatcher = { current: null };  // 内存地址: 0x001
export const __SECRET_INTERNALS = { currentDispatcher };

// Reconciler 包通过 internals 获取引用：
import * as React from 'react';
const { currentDispatcher } = React.__SECRET_INTERNALS;  // 拿到地址 0x001

// React 包设置：
React.__SECRET_INTERNALS.currentDispatcher.current = mountDispatcher;
// 修改的是地址 0x001

// Reconciler 包读取：
console.log(currentDispatcher.current);  
// 读取的也是地址 0x001，能看到修改 ✅
```

**小结：为什么 `import * as React from 'react'` 可以，但直接导入内部模块不行？**

| 对比项 | 导入内部模块 | 通过 internals 导入 |
|--------|------------|-------------------|
| **导入语句** | `import from 'react/src/...'` | `import * as React from 'react'` |
| **导入内容** | 源代码（菜谱📄） | 打包后的包（做好的菜🍖） |
| **打包行为** | 复制源代码，重新执行 | 保持外部引用 |
| **创建对象** | 创建新对象 | 引用已有对象 |
| **内存地址** | 不同地址 (0x001 和 0x002) | 同一地址 (0x001) |
| **是否共享** | ❌ 不共享 | ✅ 共享 |
| **external配置** | 不匹配，会打包 | 匹配，不打包 |

**关键点：**
1. `import from 'react'` 导入的是**整个打包好的 React 包**，里面的对象已经创建好了
2. `import from 'react/src/...'` 导入的是**源代码**，会被打包工具复制并重新执行
3. 通过 `external` 配置，`import from 'react'` 不会被打包进来，保持运行时引用
4. **只有运行时引用同一个对象，才能实现数据共享**

#### 1.7 澄清：关键区别到底是什么？

你的问题很关键！让我明确回答：

**关键区别在于：导入路径是 `'react'` 还是 `'react/xxx'`**

与使用 `import *` 还是 `import { xxx }` **无关**！

```typescript
// ✅ 这些都可以（只要从 'react' 导入）：
import * as React from 'react';           // 可以
import React from 'react';                // 可以
import { useState } from 'react';         // 可以
// 因为都匹配 external: ['react']

// ❌ 这些都不行（从 'react/xxx' 导入）：
import * as dispatcher from 'react/src/currentDispatcher';  // 不行
import dispatcher from 'react/src/currentDispatcher';       // 不行
import { resolveDispatcher } from 'react/src/currentDispatcher'; // 不行
// 因为都不匹配 external: ['react']
```

**原因分析：**

```javascript
// rollup.config.js
export default {
  external: ['react']  // ← 这个配置只匹配 'react'，不匹配 'react/xxx'
}
```

打包工具的判断逻辑：
```
遇到 import 语句时：

1. 从 'react' 导入？
   ├─ 是 → 检查是否在 external 列表
   │      ├─ 在 → 保持外部引用，不打包 ✅
   │      └─ 不在 → 打包进来
   └─ 否 → 检查是否在 external 列表
          ├─ 从 'react/src/xxx' 导入 → 'react/src/xxx' != 'react' → 不在列表 → 打包进来 ❌
          └─ 其他情况
```

**用具体例子说明：**

```typescript
// 例子 1：✅ 都可以
import * as React from 'react';
const { currentDispatcher } = React.__SECRET_INTERNALS;

import React from 'react';
const { currentDispatcher } = React.__SECRET_INTERNALS;

// 这两种写法效果一样，都是：
// - 从 'react' 包导入
// - 匹配 external 配置
// - 保持外部引用
// - 引用同一个对象


// 例子 2：❌ 都不行
import * as dispatcher from 'react/src/currentDispatcher';
import { currentDispatcher } from 'react/src/currentDispatcher';

// 这两种写法效果一样，都是：
// - 从 'react/src/xxx' 路径导入
// - 不匹配 external: ['react']
// - 会打包源代码进来
// - 创建新对象
```

**总结一句话：**

关键是 **`from 'react'` vs `from 'react/xxx'`**，而不是 `import *` vs `import {}`！

- ✅ 只要 `from 'react'`，用什么方式导入都行
- ❌ 只要 `from 'react/xxx'`（带路径），用什么方式导入都不行

2. **打包 react-dom 时，会一起处理 reconciler 的代码**
3. 当处理到 `import from 'react'` 时，使用的是 **react-dom 的 external 配置**
4. 因为 `'react'` 在 external 列表中，所以保持外部引用，不打包

**类比理解：**

- 📦 大盒子（react-dom.js）

  - 📄 reconciler（装进来的）
  - 📄 shared（装进来的）
  - 🔗 指向外面的 react（不装进来，只放个"地址"）

打包配置（external）是针对"大盒子"的：
- 告诉打包工具："遇到 'react' 时，不要装进盒子，放个地址就行"
- 这个规则对盒子里的所有东西（包括 reconciler）都有效

#### 2.2 逐行详细解释

**第 1 行：导入 Action 类型**
```typescript

```

**作用：** 导入 `Action` 类型定义

**什么是 Action？** 它是更新状态时传入的参数类型，比如：
```typescript
// Action 可以是：
// 1. 直接的新值
setState(100);  // Action = 100

// 2. 一个函数
setState(prev => prev + 1);  // Action = (prev) => prev + 1
```

所以 `Action<State>` 的定义是：
```typescript
type Action<State> = State | ((prevState: State) => State);
```

---

**第 3-5 行：定义 Dispatcher 接口**
```typescript
export interface Dispatcher {
  useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>];
}
```

**作用：** 定义"Hook 集合"的接口（契约）

**通俗理解：** Dispatcher 就像一个"工具箱的说明书"，规定了这个工具箱里应该有什么工具。

**详细拆解：**
```typescript
interface Dispatcher {
  // 工具箱里有一个工具叫 useState
  useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>]
  //        ↑   ↑                           ↑  ↑
  //        │   │                           │  └─ 返回值：[状态, 更新函数]
  //        │   └─ 参数：初始值（可以是值或函数） │
  //        └─ 泛型：状态的类型                  │
  //                                            └─ 返回类型是数组
}
```

**例子：**
```typescript
// 使用时：
const [count, setCount] = useState(0);
//     ↑      ↑           ↑         ↑
//     │      │           │         └─ 初始值
//     │      │           └─ 调用 useState
//     │      └─ Dispatch<number> 类型
//     └─ number 类型

// 类型对应：
// T = number
// initialState = 0
// 返回 [number, Dispatch<number>]
```

**为什么要定义接口？**

因为在不同阶段，useState 的实现是不同的：
```typescript
// mount 时的实现
const HooksDispatcherOnMount: Dispatcher = {
  useState: mountState  // 第一次创建状态
};

// update 时的实现
const HooksDispatcherOnUpdate: Dispatcher = {
  useState: updateState  // 更新已有状态
};
```

它们都要遵循 `Dispatcher` 接口的规定。

**作用：** 定义"更新函数"的类型

**通俗理解：** `Dispatch` 就是 `setState` 这个函数的类型定义。

**详细拆解：**
```typescript
type Dispatch<State> = (action: Action<State>) => void

//   │         └─ 状态的类型
//   └─ 类型名称
```

**例子：**
```typescript
// 定义一个 number 类型的 dispatch
```

**第 9-11 行：创建 currentDispatcher 对象**

**通俗理解：** 这是一个"插座"，可以插入不同的"插头"（不同阶段的 Dispatcher）

#### 2.3 【重要】如何理解复杂的泛型嵌套？

**你之前看不懂的根本原因：一次性理解整个类型太复杂了！**

让我教你一个**由外到内、逐层拆解、带入实例**的方法：

##### 步骤 1：先看最外层，不管内部细节

```typescript
export interface Dispatcher {
  useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>];
}
```

**第一眼只看结构：**
```
Dispatcher 是一个对象
  └─ 有个属性叫 useState
      └─ 它是个函数
```

##### 步骤 2：确定这个函数的输入和输出

```typescript
useState: <T>(参数) => 返回值
          ↑   ↑      ↑
          │   │      └─ 先看返回什么
          │   └─ 再看接收什么参数
          └─ 最后看泛型
```

**拆解：**
```typescript
useState: <T>(...) => [T, Dispatch<T>]
                      ↑
                      返回一个数组，包含两个元素
```

##### 步骤 3：带入具体例子，替换泛型

**假设我们要创建一个 number 类型的状态：**

```typescript
const [count, setCount] = useState(0);
//                                  ↑
//                                  传入 0，TypeScript 推断 T = number
```

**现在用 `T = number` 替换所有的 `T`：**

```typescript
// 原始类型：
useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>]

// 替换 T = number：
useState: (initialState: (() => number) | number) => [number, Dispatch<number>]
//        ↑              ↑                         ↑         ↑
//        │              │                         │         └─ Dispatch<number>
//        │              │                         └─ 第一个元素是 number
//        │              └─ 参数可以是函数或数字
//        └─ 不再有泛型，清晰多了！
```

##### 步骤 4：继续拆解嵌套类型

现在看 `Dispatch<number>` 是什么：

```typescript
// 原始定义：
export type Dispatch<State> = (action: Action<State>) => void;

// 替换 State = number：
type Dispatch<number> = (action: Action<number>) => void;
```

再看 `Action<number>` 是什么：

```typescript
// 原始定义（假设）：
type Action<State> = State | ((prevState: State) => State);

// 替换 State = number：
type Action<number> = number | ((prevState: number) => number);
//                    ↑        ↑
//                    │        └─ 或者是一个函数，接收旧值返回新值
//                    └─ 可以是一个数字
```

##### 步骤 5：完全展开后的类型

现在把所有嵌套都展开：

```typescript
// 最原始的复杂类型：
useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>]

// 用 T = number 替换，第一层展开：
useState: (initialState: (() => number) | number) 
          => [number, Dispatch<number>]

// Dispatch<number> 展开：
useState: (initialState: (() => number) | number) 
          => [number, (action: Action<number>) => void]

// Action<number> 展开：
useState: (initialState: (() => number) | number) 
          => [number, (action: number | ((prev: number) => number)) => void]

// 完全展开，没有任何泛型：
useState: (
  initialState: (() => number) | number
) => [
  number,
  (action: number | ((prev: number) => number)) => void
]
```

**现在完全清晰了！**

##### 步骤 6：用人话翻译

```typescript
useState 是一个函数：
  
  参数：initialState
    └─ 可以是 number
    └─ 或者是一个返回 number 的函数
  
  返回：一个数组 [状态, 更新函数]
    └─ 第一个元素：number 类型的状态值
    └─ 第二个元素：一个更新函数
        └─ 这个更新函数接收一个 action 参数
            └─ action 可以是 number
            └─ 或者是一个函数 (prev: number) => number
        └─ 更新函数没有返回值 (void)
```

##### 步骤 7：对应到实际使用

```typescript
// 使用示例：
const [count, setCount] = useState(0);
//     ↑      ↑          ↑         ↑
//     │      │          │         └─ initialState: 0 (number)
//     │      │          └─ 调用 useState
//     │      └─ setCount: (action: number | ((prev: number) => number)) => void
//     └─ count: number

// setCount 可以这样用：
setCount(100);              // action = 100 (number)
setCount(prev => prev + 1); // action = (prev: number) => number
```

##### 完整对比：一次性看 vs 逐步拆解

**一次性看（❌ 看不懂）：**
```typescript
useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>]
// 天啊！这都是什么鬼？？？
```

**逐步拆解（✅ 看懂了）：**
```typescript
第 1 步：useState 是个函数
第 2 步：T = number（用例子替换）
第 3 步：参数是 (() => number) | number
第 4 步：返回 [number, Dispatch<number>]
第 5 步：Dispatch<number> = (action: Action<number>) => void
第 6 步：Action<number> = number | ((prev: number) => number)
第 7 步：完全展开，明白了！
```

##### 核心方法总结

**理解复杂泛型的 6 步法：**

1. **只看结构** - 先不管细节，了解大致形状
2. **找输入输出** - 明确函数的参数和返回值
3. **带入实例** - 用具体类型（如 number）替换泛型 T
4. **逐层展开** - 一次展开一层嵌套
5. **翻译人话** - 用自然语言描述类型含义
6. **对应使用** - 和实际代码对照，验证理解

**关键原则：**
- ❌ 不要一次性理解所有嵌套
- ✅ 从外到内，一层一层来
- ✅ 用具体例子替换抽象泛型
- ✅ 每一层都完全理解后再进入下一层

**类比：**
理解复杂泛型就像拆俄罗斯套娃：
- 一次性看透所有层 ❌（太难了）
- 一层层打开，每层都看清楚 ✅（容易多了）

```
外层套娃：useState: <T>(...) => [T, Dispatch<T>]
  ↓ 打开
第二层：Dispatch<T> = (action: Action<T>) => void
  ↓ 打开  
第三层：Action<T> = T | ((prev: T) => T)
  ↓ 完全打开
最里层：具体的类型，没有泛型了！
```

#### 2.4 resolveDispatcher 函数详解

现在来看后半部分的核心函数：

```typescript

18   }
```

##### 第 13 行：函数签名


**拆解：**
```typescript
export const resolveDispatcher = (): Dispatcher => {
//            ↑                  ↑    ↑
//            │                  │    └─ 返回值类型：Dispatcher
//            │                  └─ 没有参数
```

**作用：** 导出一个函数，用来"获取"当前的 Dispatcher

**为什么叫 resolve（解析）？**

##### 第 14 行：获取当前 Dispatcher


**作用：** 从"容器"中取出当前的 Dispatcher

**流程：**
```typescript

  .current                 // 取出 current 属性的值
                          // 可能是 null，也可能是 Dispatcher 对象
```

**例子：**

```typescript

}
```

**作用：** 检查 dispatcher 是否为空，如果是就抛出错误

**为什么需要这个检查？**

因为只有在 React 渲染组件时，才会设置 `currentDispatcher.current`：

```typescript

```

```typescript

```

**作用：** 如果检查通过，返回找到的 Dispatcher

**此时 dispatcher 的类型：**
```typescript
// TypeScript 的类型推断：
// 第 14 行：dispatcher 可能是 Dispatcher | null
const dispatcher = currentDispatcher.current;  // Dispatcher | null

// 第 16-18 行：检查如果是 null 就抛错
if (dispatcher === null) {
  throw new Error(...);
}

// 第 19 行：能走到这里，TypeScript 知道 dispatcher 一定不是 null

```

这就是 TypeScript 的**类型收窄**（Type Narrowing）！

##### 第 22 行：默认导出


**作用：** 把 `currentDispatcher` 对象作为默认导出

**为什么要导出？**

#### 2.5 整体流程串联

现在把所有部分串起来，看完整的使用流程：

```typescript

// ============================================
// packages/react/src/index.ts
export const useState = (initialState) => {
  const dispatcher = resolveDispatcher();  // ← 调用 resolveDispatcher
  return dispatcher.useState(initialState);
};

// ============================================
// 步骤 2：resolveDispatcher 获取当前 dispatcher
// ============================================
// packages/react/src/currentDispatcher.ts
export const resolveDispatcher = () => {
  const dispatcher = currentDispatcher.current;  // ← 从容器中取出
  
  if (dispatcher === null) {
    throw new Error('hook只能在函数组件中执行');
  }
  
  return dispatcher;  // ← 返回 HooksDispatcherOnMount 或 HooksDispatcherOnUpdate
};

// ============================================
// 步骤 3：Reconciler 在渲染前设置 dispatcher
// ============================================
// packages/react-reconciler/src/fiberHooks.ts
export function renderWithHooks(wip: FiberNode) {
  // 判断是 mount 还是 update
  const current = wip.alternate;
  
  if (current !== null) {
    // update
    currentDispatcher.current = HooksDispatcherOnUpdate;  // ← 设置为 update 版本
  } else {
    // mount
    currentDispatcher.current = HooksDispatcherOnMount;  // ← 设置为 mount 版本
  }
  
  // 执行组件函数
  const Component = wip.type;
  const children = Component(props);  // ← 组件内部会调用 useState
  
  // 重置
  currentDispatcher.current = null;
}
```

```
   │
2. 执行组件函数

   │   │

   │   └─→ 执行 mountState(0)
   │
   └─→ 返回 [state, dispatch]
       │
4. 组件执行完毕
   │
```

#### 2.6 为什么要这样设计？（设计思路）

**问题：** 为什么不直接这样写？

```typescript
// ❌ 简单但错误的设计
export const useState = (initialState) => {
  return mountState(initialState);  // 直接调用 mountState
};
```

**答案：** 因为 useState 的实现在不同阶段是不同的！

| 阶段 | 要做的事 | 实现函数 |
|------|---------|----------|
| mount | 创建新状态 | mountState |
| update | 读取并更新已有状态 | updateState |

**所以需要：**

1. **一个"容器"**（currentDispatcher）来存放当前应该用的实现
2. **一个"获取器"**（resolveDispatcher）来从容器中取出实现
3. **一个"切换器"**（renderWithHooks）在不同阶段设置不同的实现

**类比：**

就像一个万能遥控器：
- 按钮（useState）永远是那个按钮
- 但根据你想控制的设备（mount/update）
- 遥控器内部会切换不同的编码方式（HooksDispatcherOnMount/OnUpdate）

**用户视角：**
```typescript
// 用户只看到一个 useState
const [count, setCount] = useState(0);

// 不用关心内部是 mount 还是 update
// 统一的接口，自动切换实现
```

**实现视角：**
```typescript
// 内部根据阶段自动切换：
currentDispatcher.current = mount ? HooksDispatcherOnMount : HooksDispatcherOnUpdate;

// mount 时：useState → resolveDispatcher → HooksDispatcherOnMount.useState → mountState
// update 时：useState → resolveDispatcher → HooksDispatcherOnUpdate.useState → updateState
```

#### 2.7 总结：currentDispatcher.ts 的作用

**这个文件做了什么？**

1. **定义类型**（Dispatcher、Dispatch）- 规定接口
2. **创建容器**（currentDispatcher）- 存放当前实现
3. **提供获取器**（resolveDispatcher）- 安全地获取实现
4. **导出给其他包使用** - 让 Reconciler 能设置，让 React 能获取

**核心价值：**

实现了一个**可切换的 Hook 实现机制**，让同一个 useState 函数在不同阶段有不同的行为。

**类比总结：**

```
currentDispatcher.ts 就像一个"智能插座"：

1. 定义了插座的接口（Dispatcher）
2. 提供了一个插座（currentDispatcher）
3. 提供了安全检查（resolveDispatcher）
4. 允许其他人切换插头（export default）

用户只管插电用，不用关心插座里装的是什么插头。
内部会根据情况自动切换合适的插头。
```

#### 3.1 useState 的导出：为什么要这样写？

```typescript
export const useState: Dispatcher['useState'] = (initialState) => {
  const dispatcher = resolveDispatcher();
  return dispatcher.useState(initialState);
};
```

##### 第一个疑问：为什么不直接实现 useState？

**你可能想：** 为什么不直接这样写？

```typescript
// ❌ 为什么不这样？
export const useState = (initialState) => {
  // 直接创建状态
  const state = initialState;
  const setState = (newValue) => { ... };
  return [state, setState];
};
```

**答案：** 因为 React 包不知道怎么实现！

**原因分析：**

```
React 包的职责：
├─ 对外提供 API（useState、useEffect 等）
├─ 定义类型（Dispatcher、Dispatch 等）
└─ 但不负责具体实现！

react-reconciler 包的职责：
├─ 实现 Hooks 的具体逻辑（mountState、updateState）
├─ 管理 Fiber 树
└─ 处理渲染流程

为什么分开？
└─ React 包可以用于不同平台（Web、Native、自定义 Renderer）
    └─ 每个平台的实现不同，但 API 相同
```

**类比理解：**

```
React 包 = 餐厅的菜单
  └─ 告诉你有哪些菜（useState、useEffect）
  └─ 但菜单不会做菜

react-reconciler 包 = 后厨
  └─ 真正做菜的地方
  └─ 根据不同情况（mount/update）用不同做法

用户点菜（调用 useState）
  ↓
菜单转发给后厨（resolveDispatcher）
  ↓
后厨做好菜（mountState/updateState）
  ↓
端上来给用户（返回 [state, setState]）
```

##### 第二个疑问：这个函数具体做了什么？

**逐行拆解：**

```typescript
export const useState: Dispatcher['useState'] = (initialState) => {
//            ↑        ↑                        ↑
//            │        │                        └─ 参数：初始值
//            │        └─ 类型：和 Dispatcher 接口中的 useState 类型一致
//            └─ 导出名称
  
  const dispatcher = resolveDispatcher();
  //    ↑            ↑
  //    │            └─ 获取当前的 dispatcher（HooksDispatcherOnMount 或 OnUpdate）
  //    └─ 局部变量，保存获取到的 dispatcher
  
  return dispatcher.useState(initialState);
  //     ↑          ↑        ↑
  //     │          │        └─ 传入初始值
  //     │          └─ 调用 dispatcher 的 useState 方法
  //     └─ 返回结果 [state, setState]
};
```

**完整流程：**

```typescript
// 1. 用户调用
const [count, setCount] = useState(0);

// 2. 进入 React 包的 useState
export const useState = (initialState) => {  // initialState = 0
  
  // 3. 获取当前 dispatcher
  const dispatcher = resolveDispatcher();
  // 假设现在是 mount 阶段
  // dispatcher = HooksDispatcherOnMount = { useState: mountState }
  
  // 4. 调用 dispatcher 的 useState
  return dispatcher.useState(initialState);
  //     = HooksDispatcherOnMount.useState(0)
  //     = mountState(0)
  //     = [0, dispatch函数]
};

// 5. 返回给用户
// count = 0
// setCount = dispatch函数
```

##### 第三个疑问：为什么类型是 `Dispatcher['useState']`？

```typescript
export const useState: Dispatcher['useState'] = ...
//                     ↑
//                     这是什么意思？
```

**解释：** 这是 TypeScript 的**索引访问类型**（Indexed Access Type）

**拆解：**

```typescript
// Dispatcher 接口定义：
interface Dispatcher {
  useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>];
}

// Dispatcher['useState'] 的意思：
// "从 Dispatcher 接口中，取出 useState 这个属性的类型"

Dispatcher['useState']
  = <T>(initialState: (() => T) | T) => [T, Dispatch<T>]
```

**为什么要这样写？**

1. **保持类型一致**
```typescript
// ✅ 使用索引访问类型
export const useState: Dispatcher['useState'] = ...

// 如果 Dispatcher 接口改了，这里自动跟着改
// 不会出现类型不匹配
```

2. **避免重复定义**
```typescript
// ❌ 不用索引访问类型
export const useState: <T>(initialState: (() => T) | T) => [T, Dispatch<T>] = ...
//                     ↑
//                     又写了一遍类型，容易出错
```

**类比：**

```
Dispatcher 接口 = 工作合同
  └─ 规定了员工要做的工作（useState、useEffect 等）

Dispatcher['useState'] = 从合同中摘出 useState 这一条
  └─ 确保实际工作（React 包的 useState）符合合同
```

#### 3.2 __SECRET_INTERNALS：这个奇怪的名字是什么？

```typescript
export const __SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED = {

```

##### 第一个疑问：为什么名字这么长这么吓人？

**答案：** 这是 React 团队故意的！

**目的：**

1. **警告开发者：** 这是内部 API，不要直接使用
2. **名字太长：** 让你打字都嫌麻烦，不想用
3. **威胁语气：** "YOU_WILL_BE_FIRED"（你会被解雇）让你害怕

**为什么不能直接用？**

```typescript

import React from 'react';
const { currentDispatcher } = React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED;
```

**那为什么要导出？**

因为 **React 内部的其他包需要访问它**（如 react-reconciler）

```typescript

import * as React from 'react';
const internals = React.__SECRET_INTERNALS_DO_NOT_USE_OR_YOU_WILL_BE_FIRED;
```

```
React 包                     shared 包                     Reconciler 包
   ↓                            ↓                              ↓
```

| 转发机制 | 不实现具体逻辑，转发给 dispatcher |
| 平台无关 | React 包不关心平台，由 reconciler 实现 |

**__SECRET_INTERNALS 的导出：**

| 作用 | 说明 |
|------|------|
| 内部通信 | 让 React 内部的包能共享数据 |
| 对象共享 | 确保所有包访问同一个 currentDispatcher |
| 警告用户 | 超长名字警告开发者不要直接使用 |
| 灵活架构 | 解耦 React 包和 reconciler 包 |

**核心思想：**

```
React 包 = 前台接待员
  └─ 接收用户请求（useState）
  └─ 转发给后厨（dispatcher）
  └─ 提供内部传话通道（__SECRET_INTERNALS）

reconciler 包 = 后厨厨师
  └─ 真正做事情的地方
  └─ 根据情况选择不同做法（mount/update）

shared 包 = 传话筒
  └─ 连接前台和后厨
  └─ 确保信息准确传递
```

```typescript
  // 4. 创建更新队列和 dispatch 函数
  const queue = createUpdateQueue();
  hook.updateQueue = queue;
  const dispatch = dispatchSetState.bind(null, currentlyRenderingFiber, queue);
  
  // 5. 返回 [state, dispatch]
  return [memoizedState, dispatch];
}
```

#### 4.9 总结：fiberHooks.ts 的核心作用

**这个文件做了什么？**

| 部分 | 作用 |
|------|------|
| 全局变量 | 记录当前渲染的 Fiber 和当前处理的 Hook |
| Hook 接口 | 定义 Hook 的数据结构（状态、更新队列、链表） |
| renderWithHooks | 渲染函数组件，管理 Hooks 生命周期 |
| HooksDispatcherOnMount | mount 阶段的 Hook 实现集合 |
| mountState | 创建新状态的具体实现（待完善） |

**核心流程：**

```
1. 准备：设置全局变量
2. 判断：mount 还是 update
3. 设置：对应的 Dispatcher
4. 执行：组件函数
5. 清理：重置全局变量
```

**关键设计：**

1. **全局变量追踪**
   - `currentlyRenderingFiber` 知道数据存哪里
   - `workInProgressHook` 知道当前处理哪个 Hook

2. **Dispatcher 切换**
   - 根据阶段自动切换实现
   - 用户无感知

3. **链表结构**
   - 多个 Hook 通过 `next` 连接
   - 按顺序执行

**类比总结：**

```
fiberHooks.ts 就像一个"演出舞台"：

1. currentlyRenderingFiber = 当前表演的演员
2. workInProgressHook = 演出进度（演到哪一幕）
3. renderWithHooks = 舞台监督
   ├─ 演出前：准备舞台（设置全局变量）
   ├─ 演出中：指挥表演（执行组件）
   └─ 演出后：清理舞台（重置变量）
4. HooksDispatcherOnMount = 首演剧本
5. mountState = 具体表演动作
```
