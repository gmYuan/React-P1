# React 更新机制实现完整指南


## 第一部分：顶层鸟瞰

### 背景问题

在 React 应用中，状态更新是最常见的操作。开发者可以通过多种方式触发更新：

- **ReactDOM.createRoot().render()** - 挂载应用时触发初始渲染
- **this.setState()** - 类组件中触发状态更新
- **useState 的 dispatch 方法** - 函数组件中触发状态更新

**核心挑战：**
- 更新可能发生于**任意组件**
- 但更新流程必须从**根节点递归**开始
- 需要一套**统一的更新机制**来兼容所有触发方式


### 宏观图

**更新机制的组成部分：**

```
┌────────────────────────────────────────────────┐
│             UpdateQueue                        │
│  ┌──────────────────────────────────────────┐ │
│  │          shared.pending                  │ │
│  │  ┌────────────────────────────────────┐  │ │
│  │  │        update (环形链表)           │  │ │
│  │  │  ┌────────┐   ┌────────┐           │  │ │
│  │  │  │ update │ → │ update │ → ...     │  │ │
│  │  │  └────────┘   └────────┘           │  │ │
│  │  └────────────────────────────────────┘  │ │
│  └──────────────────────────────────────────┘ │
└────────────────────────────────────────────────┘
```

**两个核心数据结构：**
1. **Update** - 代表更新的数据结构
2. **UpdateQueue** - 消费 update 的数据结构（队列）



### 关键技术栈

| 技术 | 说明 |
|------|------|
| **TypeScript** | 类型定义（Action、Update、UpdateQueue） |
| **环形链表** | shared.pending 使用环形链表存储多个 update |
| **泛型** | Update<State>、UpdateQueue<State> 支持不同状态类型 |
| **函数式更新** | Action 支持函数形式 `(prevState) => newState` |

### 步骤 2：创建 updateQueue.ts

**路径：** `packages/react-reconciler/src/updateQueue.ts`

```typescript
import { Action } from 'shared/ReactTypes';

export interface Update<State> {
	action: Action<State>;
}

export interface UpdateQueue<State> {
	shared: {
		pending: Update<State> | null;
	};
}

export const createUpdate = <State>(action: Action<State>): Update<State> => {
	return {
		action
	};
};

export const createUpdateQueue = <Action>() => {
	return {
		shared: {
			pending: null
		}
	} as UpdateQueue<Action>;
};

export const enqueueUpdate = <Action>(
	updateQueue: UpdateQueue<Action>,
	update: Update<Action>
) => {
	updateQueue.shared.pending = update;
};

export const processUpdateQueue = <State>(
	baseState: State,
	pendingUpdate: Update<State> | null
): { memoizedState: State } => {
	const result: ReturnType<typeof processUpdateQueue<State>> = {
		memoizedState: baseState
	};

	if (pendingUpdate !== null) {
		// baseState 1 update 2 -> memoizedState 2
		// baseState 1 update (x) => 4x -> memoizedState 4
		const action = pendingUpdate.action;
		if (action instanceof Function) {
			result.memoizedState = action(baseState);
		} else {
			result.memoizedState = action;
		}
	}

	return result;
};
```

## 第四部分：个人 Q&A

### Q1：什么是 update、什么是 UpdateQueue？为什么要引入它们？

**从一个完整的 React 更新流程说起：**

```
用户操作：
  onClick={() => setState(5)}
  
发生了什么？
  第1步：React 创建一个 update = { action: 5 }
  第2步：把这个 update 放到 UpdateQueue 里
  第3步：触发重新渲染（从根节点开始遍历 Fiber 树）
  第4步：遍历到这个组件时，读取 UpdateQueue
  第5步：计算新状态 (processUpdateQueue)
  第6步：用新状态重新渲染组件
```

**关键：**
- 步骤1和2 发生在**组件内**（比如 Counter 组件）
- 步骤3 从**根节点**开始
- 步骤4-6 又回到**这个组件**

**所以需要一个"地方"存储 update，这个地方就是 UpdateQueue！**

---

**update 和 UpdateQueue 的关系：**

| 概念 | 作用 | 类比 |
|------|------|------|
| **update** | 记录一次状态更新 | 点餐单上的一行（一道菜） |
| **UpdateQueue** | 管理多个 update | 整个点餐单（记录所有菜） |

**具体结构：**

```typescript
// update 的结构
{
  action: 5  // 或 (prevState) => prevState + 1
}

// UpdateQueue 的结构
{
  shared: {
    pending: update1 → update2 → update3  // 环形链表
  }
}
```

---

### Q2：为什么每个 FiberNode 有自己的 updateQueue，而不是共享一个全局的？

**问题场景：**

```jsx
<App>
  <Counter />  ← Counter: count = 0
  <Timer />    ← Timer: time = 100
</App>
```

现在用户同时操作：
1. 点击 Counter 按钮 → `setCount(1)`
2. 点击 Timer 按钮 → `setTime(200)`

---

**如果用一个全局 UpdateQueue：**

```typescript
全局 UpdateQueue = {
  shared: {
    pending: update1 → update2
  }
}

update1 = { action: 1 }      // 这是给 Counter 的
update2 = { action: 200 }    // 这是给 Timer 的

问题：React 怎么知道 update1 是给 Counter 的，update2 是给 Timer 的？
```

**无法区分！所有组件的更新混在一起了！**

---

**如果每个 FiberNode 有自己的 UpdateQueue：**

```typescript
CounterFiber.updateQueue = {
  shared: {
    pending: update1  // { action: 1 }
  }
}

TimerFiber.updateQueue = {
  shared: {
    pending: update2  // { action: 200 }
  }
}

✓ 清晰！每个组件的更新都在自己的队列里
✓ 遍历到 Counter 时，处理 CounterFiber.updateQueue
✓ 遍历到 Timer 时，处理 TimerFiber.updateQueue
```

---

**类比：**
- 全局 UpdateQueue = 整个餐厅只有一张点餐单（无法区分哪桌客人点了什么）
- 每个 FiberNode 有自己的 UpdateQueue = 每桌客人有自己的点餐单（清晰明了）

---

**准确的理解：**

不是"每个 FiberNode 都有 updateQueue"，而是：

| FiberNode 类型 | 有 state？ | 需要 updateQueue？ |
|---------------|-----------|-------------------|
| 函数组件（useState） | ✓ | ✓ |
| 类组件（this.state） | ✓ | ✓ |
| div、span 等 DOM 节点 | ✗ | ✗ |
| 文本节点 | ✗ | ✗ |

**只有有 state 的组件 FiberNode 才需要 updateQueue。**

### Q3：React 为什么要有两棵 Fiber 树？

**先看没有两棵树的问题：**

```
假设只有一棵树：

用户点击按钮 → setState(新值)
  ↓
React 开始遍历树，更新每个节点
  ↓
遍历到一半... 用户又点击了！
  ↓
怎么办？
  - 继续遍历旧的？但状态又变了
  - 重新开始？前面的工作白做了
  - 同时处理两个更新？树结构会乱
```

**问题：一棵树无法同时处理"当前状态"和"正在计算的新状态"**

---

**解决方案：维护两棵树**

```
树1：current（当前页面显示的）
  CounterFiber { state: 0 }  ← 页面显示 0

树2：workInProgress（正在计算新状态的）
  CounterFiber { state: 1 }  ← 正在算出新状态 1
```

**类比：**
- current 树 = 正在放映的电影（观众正在看）
- workInProgress 树 = 正在剪辑的下一帧（观众看不到）

---

### Q4：为什么 UpdateQueue 使用 `shared.pending` 而不是直接用 `pending`？

**核心问题：为什么不能在交换时直接复制？**

你可能会想：既然两棵树会交换，那为什么不这样做？

```typescript
// 看起来可行的方案：
workInProgress.updateQueue = { pending: update1, update2 }
current.updateQueue = { pending: null }

// 完成渲染后交换：
current（原来的 workInProgress）.updateQueue = { pending: update1, update2 }
workInProgress（原来的 current）.updateQueue = current.updateQueue  // 复制过来
```

**答案：这个方案在 React 的可中断渲染场景下会丢失更新！**

---

**关键：React 的渲染可以被中断、重试、暂停**

```
情况A：不共享（各自有 pending）

[首次渲染]
current.updateQueue → qC { pending: null }
workInProgress.updateQueue → qW1 { pending: null }

[用户触发更新]
workInProgress.updateQueue → qW1 { pending: U1 }

[渲染中断！React 重建 WIP2]
current.updateQueue → qC { pending: null }
workInProgress.updateQueue → qW2 { pending: null }  ← U1 丢了！

问题：U1 还在旧对象 qW1 里，新的 WIP2 指向新对象 qW2，看不到 U1
```

---

**时间线模拟（展示如何丢失更新）：**

```
T1：开始渲染
    current.updateQueue.pending = null
    WIP1.updateQueue.pending = null

T2：用户触发更新 U1
    WIP1.updateQueue.pending = [U1]
    current.updateQueue.pending = null

T3：渲染到一半被中断（第一次中断）
    React 可能丢弃 WIP1，准备重试

T4：重试，创建 WIP2（从 current 复制）
    WIP2.updateQueue.pending = null  ← 只复制了 current 的值
    U1 还在 WIP1 里，但 WIP1 已被丢弃

T5：再次中断，创建 WIP3
    WIP3.updateQueue.pending = null  ← 又是从 current 复制
    U1 永远丢失

T6：最终提交
    页面没有 U1 的效果
```

---

**情况B：用 `shared.pending`（共享）**

```
[首次渲染]
current.updateQueue ──┐
                      ↓
                  qs { shared: { pending: null } }  ← 同一个对象
                      ↑
workInProgress.updateQueue ──┘

[用户触发更新]
qs.shared.pending = U1  ← 写到共享对象里

[中断后重建 WIP2]
current.updateQueue ──┐
                      ↓
                  qs { shared: { pending: U1 } }  ← 仍然是同一个对象
                      ↑
workInProgress.updateQueue ──┘

✓ 无论重建多少次，都指向同一个 qs
✓ U1 永远不会丢失
```

---

**一句话记忆法：**

| 方案 | 特点 | 结果 |
|------|------|------|
| **各自的 pending** | 两只独立便签盒 | 容易在重建/交换时找错盒子，丢失更新 |
| **shared.pending** | 一只公共便签盒 | 更新随时写入，任何一代树都能读到 |

---

**类比：**
- 不用 `shared`：每个草稿本都有自己的便签盒，换草稿本时容易丢便签
- 用 `shared`：把便签盒抽出来放公共文件柜，无论换几次草稿本，便签都在柜子里

---

**为什么 React 需要可中断渲染？**

React 18 的并发特性允许：
- 高优先级更新可以打断低优先级渲染
- 用户交互（点击）可以打断数据加载
- 保证页面不卡顿

这就要求更新队列必须在中断/重试时保持一致，所以必须用 `shared` 共享。

### Q5：`as UpdateQueue<Action>` 是什么意思，为什么要这么写？

**先看代码：**

```typescript
export const createUpdateQueue = <Action>() => {
  return {
    shared: {
      pending: null
    }
  } as UpdateQueue<Action>;
};
```

---

**第一步：`as` 是什么**

`as` 叫做"类型断言"，你可以理解成"我明确告诉 TypeScript 这个东西的类型是什么"。

**类比：**  
你写了一张便签，上面只写了"共享，待办：空"。  
TypeScript 看到后说："我不知道这是什么格式的便签啊，可能只是普通便条？"  
你用 `as UpdateQueue<Action>` 相当于贴个标签："这就是标准的更新队列格式！"

---

**第二步：为什么需要**

看你返回的对象：

```typescript
{
  shared: {
    pending: null
  }
}
```

TypeScript 自己推断出的类型是：

```typescript
{
  shared: {
    pending: null  // 类型是 null
  }
}
```

但 `UpdateQueue<Action>` 接口要求的类型是：

```typescript
{
  shared: {
    pending: Update<Action> | null  // 类型可以是 Update 或 null
  }
}
```

**问题：**  
TypeScript 认为你这个 `null` 太"窄"了（只能是 null），不符合接口定义（应该能装 Update 或 null）。

---

**第三步：如果不用 `as` 会怎么样？**

TypeScript 会报错：

```typescript
// 如果不用 as
export const createUpdateQueue = <Action>() => {
  return {
    shared: {
      pending: null
    }
  };
  // ❌ 错误：返回类型不匹配
};
```

**具体报错信息可能是：**

```
Type '{ shared: { pending: null; }; }' is not assignable to type 'UpdateQueue<Action>'.
  Types of property 'shared' are incompatible.
    Type '{ pending: null; }' is not assignable to type '{ pending: Update<Action> | null; }'.
```

---

**第四步：核心理解**

`as UpdateQueue<Action>` 的作用：

| 作用 | 说明 |
|------|------|
| **扩展类型** | 告诉 TS："这个 null 将来可以是 Update 或 null" |
| **类型安全** | 确保返回值符合 UpdateQueue 接口 |
| **避免报错** | 让 TS 相信你的类型是对的 |

**类比：**
- 你造了一个空盒子（只有 null）
- TypeScript 说："这盒子太小了，装不下 Update"
- 你用 `as` 说："这是标准大小的盒子，现在是空的，以后能装 Update"

### Q6：`ReturnType<typeof processUpdateQueue<State>>` 是什么意思，为什么要这么写？

**先看代码：**

```typescript
export const processUpdateQueue = <State>(
  baseState: State,
  pendingUpdate: Update<State> | null
): { memoizedState: State } => {
  const result: ReturnType<typeof processUpdateQueue<State>> = {
    memoizedState: baseState
  };
  // ...
};
```

---

**第一步：拆解这个类型**

这个类型由三部分组成：

| 部分 | 含义 |
|------|------|
| `ReturnType<...>` | TypeScript 工具类型，用来获取函数的返回值类型 |
| `typeof processUpdateQueue` | 获取 `processUpdateQueue` 函数本身的类型 |
| `<State>` | 传入泛型参数 |

**组合起来：**  
"获取 `processUpdateQueue<State>` 这个函数的返回值类型"

---

**第二步：先理解 `typeof` 在 TypeScript 类型系统中的作用**

在 TypeScript 的类型系统中，`typeof` 用来**获取某个变量/函数的类型定义**。

**例子1：获取变量的类型**

```typescript
const user = {
  name: "张三",
  age: 25
};

// typeof user = { name: string; age: number }
type UserType = typeof user;
```

**例子2：获取函数的类型**

```typescript
function add(a: number, b: number): number {
  return a + b;
}

// typeof add = (a: number, b: number) => number
type AddType = typeof add;
```

**类比：**
- `typeof` = 照相机
- `add` 函数 = 一个人
- `typeof add` = 给这个人拍了张"类型照片"，照片上写着函数的完整签名

---

**第三步：理解 `ReturnType<...>`**

`ReturnType` 是 TypeScript 内置的工具类型，作用是：**从函数类型中提取返回值类型**。

**例子：**

```typescript
function add(a: number, b: number): number {
  return a + b;
}

// 第1步：typeof add = (a: number, b: number) => number
// 第2步：ReturnType<typeof add> = number（提取返回值类型）

type AddReturnType = ReturnType<typeof add>;
// AddReturnType = number
```

**类比：**
- 函数类型 = 一份简历（包含参数、返回值等信息）
- `ReturnType` = 一个筛选器，只看"返回值类型"那一栏

---

**第四步：组合理解**

```typescript
const result: ReturnType<typeof processUpdateQueue<State>>
```

**分解步骤：**

```typescript
// 第1步：processUpdateQueue<State>
// 传入具体的泛型参数 State

// 第2步：typeof processUpdateQueue<State>
// 获取这个函数的类型定义
// = (baseState: State, ...) => { memoizedState: State }

// 第3步：ReturnType<typeof processUpdateQueue<State>>
// 从函数类型中提取返回值类型
// = { memoizedState: State }
```

**所以这行代码等价于：**

```typescript
const result: { memoizedState: State } = {
  memoizedState: baseState
};
```

---

**第五步：为什么要这么绕着写？**

**好处1：保持类型一致性**

如果将来修改了函数返回值：

```typescript
// 修改后：新增了一个字段
processUpdateQueue<State>(...): { 
  memoizedState: State,
  hasMore: boolean  // ← 新增
}
```

**用 ReturnType 的好处：**
```typescript
const result: ReturnType<typeof processUpdateQueue<State>> = {
  memoizedState: baseState,
  hasMore: false  // ← 必须加上，否则报错
};
// ✓ 自动同步，强制你补齐字段
```

---

**好处2：避免重复定义**

```typescript
// 不用 ReturnType：重复写了两次
processUpdateQueue<State>(...): { memoizedState: State } {
  const result: { memoizedState: State } = { ... };  // 重复！
}

// 用 ReturnType：只写一次
processUpdateQueue<State>(...): { memoizedState: State } {
  const result: ReturnType<typeof processUpdateQueue<State>> = { ... };
  // 自动从函数签名推导
}
```

---

**核心理解：**

| 写法 | 优点 | 缺点 |
|------|------|------|
| `const result: { memoizedState: State }` | 简单直接 | 函数返回值变化时不会自动同步 |
| `const result: ReturnType<typeof processUpdateQueue<State>>` | 自动同步，类型安全 | 看起来复杂 |

---

**最简单的类比：**

想象一个工厂生产线：

1. **`processUpdateQueue`** = 一条生产线（函数）
2. **`typeof processUpdateQueue`** = 这条生产线的说明书（函数类型）
3. **`ReturnType<typeof processUpdateQueue>`** = 从说明书中找到"产品规格"那一页（返回值类型）

**所以：**
```typescript
const result: ReturnType<typeof processUpdateQueue<State>>
```

意思是："result 这个变量的类型，就是 processUpdateQueue 函数产出的那个产品的类型"。

**一句话总结：**  
这是 TypeScript 的"单一数据源"思想——返回值类型只在函数签名定义一次，内部变量通过 `ReturnType` 自动推导，保证类型始终一致。
