# React 首屏渲染实现完整指南

## 第一部分：顶层鸟瞰

### 背景问题

在 React 应用中，首屏渲染是应用启动的关键流程。开发者通过以下方式挂载应用：

```javascript
ReactDOM.createRoot(rootElement).render(<App/>)
```

**核心挑战：**
- 如何连接 DOM 容器和 React Fiber 树
- 如何触发首次渲染
- 如何将更新机制接入到首屏渲染中

### 宏观图

**首屏渲染的数据结构关系：**

```
ReactDOM.createRoot(rootElement).render(<App/>)
           ↓
    FiberRootNode
           ↓
     current ↓↑ stateNode
           ↓
    hostRootFiber
           ↓
      child ↓↑ return
           ↓
         App
```

**核心组件：**
1. **FiberRootNode** - 整个应用的根节点
2. **hostRootFiber** - Fiber 树的根节点
3. **Container** - DOM 容器

### 关键技术栈

| 技术 | 说明 |
|------|------|
| **TypeScript** | 类型定义（Container、FiberRootNode） |
| **Fiber 架构** | hostRootFiber 作为 Fiber 树的根节点 |
| **更新机制** | 通过 UpdateQueue 触发首次渲染 |
| **调度机制** | scheduleUpdateOnFiber 调度更新 |

### 步骤 5：创建并实现 fiberReconciler.ts

**路径：** `packages/react-reconciler/src/fiberReconciler.ts`

**说明：** 创建 Reconciler 对外接口，提供容器创建和更新方法

```typescript
import { Container } from 'hostConfig';
import { ReactElementType } from 'shared/ReactTypes';
import { FiberNode, FiberRootNode } from './fiber';
import { createUpdate, createUpdateQueue, enqueueUpdate, UpdateQueue } from './updateQueue';
import { scheduleUpdateOnFiber } from './workLoop';
import { HostRoot } from './workTags';

export function createContainer(container: Container) {
  const hostRootFiber = new FiberNode(HostRoot, {}, null);
  const root = new FiberRootNode(container, hostRootFiber);
  hostRootFiber.updateQueue = createUpdateQueue();
  return root;
}

export function updateContainer(
  element: ReactElementType | null,
  root: FiberRootNode
) {
  const hostRootFiber = root.current;
  const update = createUpdate<ReactElementType | null>(element);
  enqueueUpdate(
    hostRootFiber.updateQueue as UpdateQueue<ReactElementType | null>,
    update
  );
  scheduleUpdateOnFiber(hostRootFiber);
  return element;
}
```

### Q1: 刚刚的改动主要做了哪些事情，实现了哪些功能？项目的入口是什么？

我们实现了 React 应用的"启动流程" —— 从创建根节点到触发渲染的完整链路。

就像盖楼一样：打好地基（FiberRootNode）→ 建立结构（hostRootFiber）→ 安装电梯（createContainer/updateContainer）→ 启动调度系统（scheduleUpdateOnFiber）。

#### 完整的调用链

```
【未来会实现】
ReactDOM.createRoot(container)
    ↓
【当前入口 ✅】
createContainer(container)
    ↓ 创建数据结构
FiberRootNode + hostRootFiber
    ↓
【当前入口 ✅】
updateContainer(<App/>, root)
    ↓ 准备更新
createUpdate + enqueueUpdate
    ↓
scheduleUpdateOnFiber(hostRootFiber)
    ↓
markUpdateFromFiberToRoot(找到 FiberRootNode)
    ↓
renderRoot(开始渲染)
    ↓
prepareFreshStack(创建 workInProgress 树)
    ↓
workLoop(遍历 Fiber 树)
```

#### 总结

我们实现了 React 从"接收 JSX"到"准备渲染"的整个启动流程。当前还没有真正操作 DOM（那是 `beginWork` 和 `completeWork` 后续要做的工作），但已经搭建好了整个架构的骨架。

### Q2.1: createContainer 函数做了什么，有什么作用？

#### 函数作用

**createContainer**
- **作用**：创建 React 应用的根节点结构，建立 DOM 容器和 Fiber 树的连接
- **产物**：一个 `FiberRootNode` 对象（包含 container、hostRootFiber 及其双向连接关系）

#### 代码拆解

```typescript
export function createContainer(container: Container) {
  // 步骤1：创建 Fiber 树的根节点
  const hostRootFiber = new FiberNode(HostRoot, {}, null);
  
  // 步骤2：创建应用根节点，建立双向连接
  const root = new FiberRootNode(container, hostRootFiber);
  //  ↓ FiberRootNode 构造函数内部做了：
  //    - root.container = container
  //    - root.current = hostRootFiber
  //    - hostRootFiber.stateNode = root
  
  // 步骤3：为 hostRootFiber 创建更新队列
  hostRootFiber.updateQueue = createUpdateQueue();
  
  // 步骤4：返回应用根节点
  return root;
}
```

#### 执行结果

**输入：** `container`（DOM 元素，如 `<div id="root">`）

**输出：** `FiberRootNode` 对象，包含：
- `root.container` → 指向 DOM 容器
- `root.current` → 指向 hostRootFiber
- `hostRootFiber.stateNode` → 指向 root（双向连接）
- `hostRootFiber.updateQueue` → 更新队列（用于管理状态更新）

#### 数据结构关系图

```
      FiberRootNode (root)
      ┌─────────────────────┐
      │ container: div#root │ ← 指向真实 DOM
      │ current: ──────────┐│
      │ finishedWork: null ││
      └────────────────────┼┘
                           ↓
                    hostRootFiber (FiberNode)
                    ┌──────────────────────┐
                    │ tag: HostRoot        │
                    │ stateNode: ────────┐ │ ← 指回 root（双向连接）
                    │ updateQueue: {...} │ │
                    └────────────────────┼─┘
                                         ↓
                                   回指 root
```

#### 示例

```typescript
const divElement = document.getElementById('root');
const root = createContainer(divElement);

// 此时得到：
root.container === divElement           // ✅ true
root.current.tag === HostRoot          // ✅ true
root.current.stateNode === root        // ✅ true（双向连接）
root.current.updateQueue !== null      // ✅ true
```

#### 补充："建立 DOM 容器和 Fiber 树的连接" 是什么意思？

**两个世界：**

```
DOM 世界（真实世界）         Fiber 世界（React 内部）
<div id="root">         →    FiberRootNode
                                  ↓
                            hostRootFiber
                                  ↓
                              AppFiber
```

**连接的本质：两个指针**

```
DOM 容器（真实 HTML）
       ↓ 
   container
       ↓
┌──────────────────┐
│  FiberRootNode   │
│  container: div  │ ← 指针1：React 能找到真实 DOM（知道渲染到哪里）
│  current: ───┐   │ ← 指针2：React 能找到 Fiber 树入口（知道从哪开始工作）
└──────────────┼───┘
               ↓
        hostRootFiber
```

**为什么需要这个连接？**

1. **React 需要知道"渲染到哪里"**
   ```typescript
   // 最终 DOM 要挂载到这里
   root.container.appendChild(appDOM);
   ```

2. **React 需要知道"从哪里开始遍历"**
   ```typescript
   // 开始渲染时，从 Fiber 树的根节点开始
   const fiber = root.current;  // 拿到 hostRootFiber
   workLoop(fiber);  // 开始遍历
   ```

**一句话总结：** 让 React 的内部数据结构（Fiber 树）知道要渲染到哪个真实 DOM 元素上，同时也知道从哪个 Fiber 节点开始工作。

### Q2.2: FiberNode 和 FiberRootNode 有什么区别，为什么要分成 2 个类？

#### 核心区别对比

| 维度 | FiberNode | FiberRootNode |
|------|-----------|---------------|
| **身份** | 树中的"节点" | 整个应用的"管理者" |
| **数量** | 有多少个组件就有多少个 | 整个应用只有 1 个 |
| **职责** | 代表具体的 React 组件（div、App、Button 等） | 管理整个应用的状态和容器 |
| **结构** | 通过 child、sibling、return 连接成树 | 不形成树，只是一个管理对象 |
| **字段数量** | ~15 个字段（type、props、state、child 等） | 只有 3 个字段（container、current、finishedWork） |

#### 类比理解

**公司组织架构：**

```
FiberRootNode = 公司总部大楼
    ↓
FiberNode = 员工
    ├─ 员工A（总经理）  ← hostRootFiber
    ├─ 员工B（部门经理）
    ├─ 员工C（普通员工）
    └─ ...
```

- **FiberRootNode（总部大楼）**：
  - 只有 1 栋楼
  - 管理整个公司的运作
  - 记录"当前在职员工"（current）
  - 记录"办公地址"（container）

- **FiberNode（员工）**：
  - 有很多员工
  - 每个员工有自己的职责、上司、下属
  - 形成组织架构树

#### 代码对比

**FiberNode（节点类）：**
```typescript
export class FiberNode {
  // 组件信息
  type: any;              // 组件类型（div、App、Button）
  tag: WorkTag;           // 节点类型（HostRoot、HostComponent）
  props: Props;           // 组件属性
  state: any;             // 组件状态
  
  // 树结构
  child: FiberNode;       // 子节点
  sibling: FiberNode;     // 兄弟节点
  return: FiberNode;      // 父节点
  
  // 工作单元
  alternate: FiberNode;   // 对应的另一棵树的节点
  updateQueue: unknown;   // 更新队列
  // ... 更多字段
}
```

**FiberRootNode（管理类）：**
```typescript
export class FiberRootNode {
  container: Container;        // DOM 容器（渲染到哪里）
  current: FiberNode;          // 当前的 Fiber 树根节点
  finishedWork: FiberNode;     // 完成工作的 Fiber 树根节点
}
```

#### 为什么要分成 2 个类？

**原因1：职责不同**

```
FiberRootNode 的职责：
├─ 记录 DOM 容器在哪里
├─ 管理当前显示的 Fiber 树
└─ 管理即将显示的 Fiber 树

FiberNode 的职责：
├─ 代表一个具体的组件
├─ 存储组件的 props、state
└─ 连接父子兄弟形成树结构
```

**原因2：数据结构不同**

```
FiberRootNode：
- 不需要 child、sibling、return（它不是树的一部分）
- 不需要 props、state（它不是组件）
- 只需要 container、current、finishedWork

FiberNode：
- 需要 child、sibling、return（形成树）
- 需要 props、state（组件数据）
- 需要 alternate、updateQueue（渲染机制）
```

**原因3：生命周期不同**

```
FiberRootNode：
- 应用启动时创建 1 次
- 应用运行期间一直存在
- 应用销毁时才销毁

FiberNode：
- 每次渲染都可能创建新的
- 组件卸载时会被销毁
- 可以被复用（通过 alternate）
```

#### 数据结构关系

```
整个应用只有 1 个 FiberRootNode
           ↓
    FiberRootNode
    ┌─────────────────┐
    │ container: div  │
    │ current: ───┐   │
    └─────────────┼───┘
                  ↓
           hostRootFiber (FiberNode)
                  ↓
              AppFiber (FiberNode)
                  ↓
            divFiber (FiberNode)
               ↙     ↘
         h1Fiber   pFiber
            ↓         ↓
         textFiber textFiber
         
每个组件都有一个 FiberNode
```

#### 一句话总结

**FiberRootNode** 是整个应用的"管理者"（只有1个），负责管理容器和 Fiber 树；  
**FiberNode** 是树中的"节点"（有很多个），负责代表具体的组件。

#### 补充：为什么需要创建 FiberRootNode 和 hostRootFiber 两个对象实例？

从对象实例角度看，为什么不能只用 1 个对象？

**❌ 方案1：只用 FiberRootNode**
```typescript
const root = new FiberRootNode(container);
// 问题：FiberRootNode 不是树节点，没有 child、sibling、return
// 无法遍历！workLoop 怎么开始？
```

**❌ 方案2：只用 FiberNode**
```typescript
const root = new FiberNode(HostRoot, {}, null);
// 问题：FiberNode 没有 container 字段
// 最后渲染完，往哪里挂载 DOM？
```

**✅ 方案3：用 2 个对象（当前方案）**
```typescript
const hostRootFiber = new FiberNode(HostRoot, {}, null);  // 负责遍历
const root = new FiberRootNode(container, hostRootFiber);  // 负责管理
// 各司其职，完美配合
```

**它们各自解决什么问题：**

```
hostRootFiber 的作用：
├─ 作为 Fiber 树的根节点（可以遍历）
├─ 有 child 指向 App 组件
└─ workLoop 从这里开始

FiberRootNode 的作用：
├─ 记住 DOM 容器在哪里（container）
├─ 记住 Fiber 树的入口在哪里（current）
└─ 管理渲染完成的结果（finishedWork）
```

**总结：** hostRootFiber 是"可以遍历的树的入口"，FiberRootNode 是"不参与遍历的管理者"。两个对象分工明确，缺一不可。

#### 补充：如何区分 rootFiber 和 fiberRoot？

面试或讨论时，经常会被问到这两个简称的区别。记住一个技巧：

**技巧：看名词位置（后面的词是主体）**

```
rootFiber = root + Fiber
   ↑           ↑      ↑
  修饰语      形容词  名词
  
"是一个 Fiber，这个 Fiber 是 root 类型的"

fiberRoot = fiber + Root  
   ↑          ↑       ↑
  修饰语     形容词   名词
  
"是一个 Root，这个 Root 管理 fiber"
```

**口诀：后面的词是主体！**
- `rootFiber` → 是个 **Fiber**（树节点，类型是 FiberNode）
- `fiberRoot` → 是个 **Root**（管理者，类型是 FiberRootNode）

**关系图：**
```
fiberRoot（老板）
    ↓ current
rootFiber（员工A号）
    ↓ child
AppFiber（员工B号）
```

- **fiberRoot**：不干活，只管人（不是树的一部分）
- **rootFiber**：干活的第一个员工（树的根节点）

#### Q2.3: FiberRootNode 里的 finishedWork 是干什么的，有什么作用？

##### 作用

**finishedWork** 是一个"暂存区"，用来保存**已经构建完成、准备提交到屏幕的 Fiber 树**。

##### 类比理解

想象你在餐厅点餐：

```
current（当前菜品）     = 你正在吃的菜
workInProgress（后厨）  = 厨师正在做的新菜
finishedWork（传菜区）  = 做好了，等服务员端上来

流程：
1. 你点了新菜（触发更新）
2. 厨师开始做菜（构建 workInProgress 树）
3. 菜做好了，放在传菜区（workInProgress → finishedWork）
4. 服务员端上桌（finishedWork 提交到 DOM）
5. 你开始吃新菜（finishedWork → current）
```

##### React 的工作流程

```
阶段1：开始渲染
┌─────────────┐
│   current   │ ← 屏幕显示的是这棵树
└─────────────┘
      ↓ 复制
┌─────────────┐
│    WIP      │ ← 开始构建新树
└─────────────┘

阶段2：渲染完成
┌─────────────┐
│   current   │ ← 屏幕还是显示旧的
└─────────────┘

┌─────────────┐
│    WIP      │ ← 构建完成！
└─────────────┘
      ↓ 赋值
┌─────────────┐
│finishedWork │ ← 暂存在这里 🔴
└─────────────┘

阶段3：提交到 DOM
┌─────────────┐
│finishedWork │ → 应用到真实 DOM
└─────────────┘
      ↓ 切换
┌─────────────┐
│   current   │ ← 现在指向新树
└─────────────┘
```

##### 为什么需要 finishedWork？

**原因1：分离"渲染"和"提交"两个阶段**

```typescript
// 渲染阶段（可以中断）
renderRoot(root) {
  // 构建 workInProgress 树
  workLoop();
  
  // 完成后保存到 finishedWork
  root.finishedWork = workInProgress;
}

// 提交阶段（不可中断）
commitRoot(root) {
  // 从 finishedWork 取出来
  const finishedWork = root.finishedWork;
  
  // 一次性提交到 DOM
  commitMutations(finishedWork);
  
  // 切换 current 指针
  root.current = finishedWork;
  
  // 清空
  root.finishedWork = null;
}
```

**原因2：避免提交时出现新更新的混乱**

```
假设没有 finishedWork：

时刻1: 开始渲染，workInProgress 正在构建
时刻2: 渲染完成，准备提交
时刻3: 突然来了新更新！workInProgress 被重新创建
时刻4: 提交阶段：❌ 糟糕！workInProgress 已经不是刚才那个了

有了 finishedWork：

时刻1: 开始渲染，workInProgress 正在构建
时刻2: 渲染完成，保存到 finishedWork
时刻3: 突然来了新更新！workInProgress 被重新创建
时刻4: 提交阶段：✅ 没问题！提交的是 finishedWork，不受影响
```

##### 没有它会怎么样？

**问题1：提交时机不明确**
```typescript
// 没有 finishedWork
renderRoot(root) {
  workLoop();
  // 完成了？还是没完成？不知道！
  // workInProgress 可能随时被新更新覆盖
}
```

**问题2：并发更新会混乱**
```typescript
// 场景：
// 1. 用户点击按钮A → 开始渲染更新A
// 2. 渲染到一半，用户又点击按钮B → 更新A被打断
// 3. 更新B开始渲染
// 4. 更新A想恢复渲染 → ❌ workInProgress 已经被B占用了！

// 有了 finishedWork，可以这样：
// 1. 更新A完成 → 保存到 finishedWork
// 2. 更新B开始 → 创建新的 workInProgress
// 3. 提交更新A → 从 finishedWork 取出来提交
// 4. 继续构建更新B → 不冲突！
```

**问题3：无法回滚**
```typescript
// 如果渲染过程中发现错误：
// 有 finishedWork → 不提交就行，current 还是旧的 ✅
// 没有 finishedWork → workInProgress 直接变 current，无法回滚 ❌
```

##### 代码示例

```typescript
// 完整流程（简化版）
function performWork(root: FiberRootNode) {
  // 1. 渲染阶段：构建 workInProgress 树
  renderRoot(root);
  
  // 2. 保存到 finishedWork
  root.finishedWork = workInProgressRoot;
  
  // 3. 提交阶段：应用到 DOM
  commitRoot(root);
}

function commitRoot(root: FiberRootNode) {
  const finishedWork = root.finishedWork;
  
  if (finishedWork === null) {
    return; // 没有完成的工作，直接返回
  }
  
  // 提交变更到 DOM
  commitMutationEffects(finishedWork);
  
  // 切换 current 指针
  root.current = finishedWork;
  
  // 清空 finishedWork
  root.finishedWork = null;
}
```

##### 一句话总结

**finishedWork** 是"渲染完成"和"提交DOM"之间的**中转站**，确保渲染和提交两个阶段互不干扰，同时支持并发更新和错误回滚。

#### Q2.4: 为什么要创建 FiberRootNode 和 hostRootFiber 的双向关系？

##### 双向关系回顾

在 Q2.1 中，我们看到了双向连接的存在：

```typescript
// FiberRootNode 构造函数
constructor(container: Container, hostRootFiber: FiberNode) {
  this.current = hostRootFiber;        // ← 方向1：root → fiber
  hostRootFiber.stateNode = this;      // ← 方向2：fiber → root
}
```

```
   FiberRootNode
   ┌──────────┐
   │ current ─┼──→ hostRootFiber
   └──────────┘      │
        ↑            │
        └─ stateNode ┘
```

但**为什么要建立双向关系呢？**

##### 为什么需要双向关系？

**场景1：从 FiberRootNode 找到 Fiber 树（root → fiber）**

```typescript
// 当 React 要开始渲染时
function prepareFreshStack(root: FiberRootNode) {
  // 需要从 FiberRootNode 找到 Fiber 树的入口
  const hostRootFiber = root.current;  // ✅ 通过 current 指针
  workInProgress = createWorkInProgress(hostRootFiber, {});
}
```

**用途：** 启动渲染时，从管理者（FiberRootNode）找到工作起点（hostRootFiber）

**场景2：从 Fiber 节点找到 FiberRootNode（fiber → root）**

```typescript
// 当任意组件触发更新时
function scheduleUpdateOnFiber(fiber: FiberNode) {
  // 需要从任意 Fiber 节点找到 FiberRootNode
  const root = markUpdateFromFiberToRoot(fiber);  // ← 怎么找？
  renderRoot(root);
}

function markUpdateFromFiberToRoot(fiber: FiberNode) {
  let node = fiber;
  // 向上遍历，找到 hostRootFiber
  while (node.return !== null) {
    node = node.return;
  }
  // 通过 stateNode 拿到 FiberRootNode
  return node.stateNode;  // ✅ 通过 stateNode 指针
}
```

**用途：** 任意组件更新时，从 Fiber 节点（可能在树的深处）找回管理者（FiberRootNode）

##### 类比理解

**公司员工和总部的关系：**

```
总部大楼（FiberRootNode）
    ↓ 有员工花名册（current）
员工A（hostRootFiber）
    ↓ 有工作牌（stateNode）
    工作牌上印着：所属总部大楼地址
```

**为什么需要双向？**

1. **总部找员工**（root → fiber）
   - 总部要下达任务 → 翻花名册找到员工A
   - `root.current` = 花名册

2. **员工找总部**（fiber → root）
   - 员工遇到问题要汇报 → 看工作牌找到总部
   - `fiber.stateNode` = 工作牌

##### 没有双向关系会怎么样？

**❌ 只有单向（root → fiber）：**
```typescript
// 只有 root.current，没有 fiber.stateNode

// 问题：组件更新时找不到 root
function scheduleUpdateOnFiber(fiber: FiberNode) {
  const root = ???  // ❌ 怎么找？只能向上遍历到 hostRootFiber
  // 但遍历到 hostRootFiber 后，怎么拿到 FiberRootNode？
  // 没有 stateNode 指针，拿不到！
}
```

**❌ 只有单向（fiber → root）：**
```typescript
// 只有 fiber.stateNode，没有 root.current

// 问题：渲染时找不到 Fiber 树入口
function prepareFreshStack(root: FiberRootNode) {
  const fiber = ???  // ❌ 怎么找？root 上没有 current 指针
}
```

##### 实际使用场景

**完整流程（修正版）：**

```
用户点击 Button 组件
    ↓
Button 的 FiberNode 触发更新
    ↓ 调用 scheduleUpdateOnFiber(buttonFiber)
    ↓
向上遍历：ButtonFiber → divFiber → AppFiber → hostRootFiber
    ↓ markUpdateFromFiberToRoot 找到 hostRootFiber
    ↓
通过 hostRootFiber.stateNode 找到 FiberRootNode  ← 🔴 用到反向指针
    ↓
调用 renderRoot(fiberRootNode)
    ↓ 内部调用 prepareFreshStack(root)
    ↓
通过 root.current 拿到 hostRootFiber  ← 🔴 用到正向指针
    ↓
基于 hostRootFiber 创建新的 workInProgress 树
    ↓
遍历新树，重新渲染
```

##### 为什么组件更新时要向上找 root？

**核心原因：React 的更新必须从根节点开始，确保全局一致性。**

**原因1：需要检查父组件是否也受影响**

```jsx
function App() {
  const [theme, setTheme] = useState('light');
  return (
    <div className={theme}>  {/* 父组件的 state 影响了 div */}
      <Button />              {/* Button 触发更新 */}
    </div>
  );
}

// 如果只更新 Button：
// ❌ App 的 theme 变了，div 的 className 没更新

// 从 root 开始遍历：
// ✅ 依次检查 App → div → Button，都能正确更新
```

**原因2：调度系统在 FiberRootNode 上**

```typescript
// 调度相关信息都在 FiberRootNode 上：
FiberRootNode {
  container: Container;     // 渲染到哪里
  current: FiberNode;       // 当前树（用于 diff）
  finishedWork: FiberNode;  // 完成的树
  // 真实 React 还有：
  // - 优先级队列
  // - 过期时间
  // - 调度任务
}
```

**如果不找到 root：**
- 不知道往哪个 container 渲染
- 没有 current 树，无法 diff
- 无法统一调度多个更新

**原因3：支持批量更新**

```typescript
// 场景：用户快速点击 3 次按钮
onClick() {
  setState(1);  // 更新1 → 找到 root
  setState(2);  // 更新2 → 找到 root（同一个）
  setState(3);  // 更新3 → 找到 root（同一个）
}

// 如果就地更新：渲染 3 次 ❌
// 找到 root 统一调度：合并更新，只渲染 1 次 ✅
```

##### 类比理解

**公司的请假流程：**

```
员工A（Button）要请假
    ↓ 为什么不能自己决定？
因为需要：
├─ 告诉主管（父组件）调配人手
├─ 通知人事部（FiberRootNode）更新考勤
├─ 检查是否影响项目进度（其他组件）
└─ 统一安排（批量处理）

所以必须：
员工A → 主管 → 部门经理 → 总经理 → 人事部（FiberRootNode）
       ↑ 向上找
```

##### 双向关系的价值总结

```
正向（root → fiber）：
- 渲染时找到工作起点
- 知道从哪里开始遍历

反向（fiber → root）：
- 更新时找到调度中心
- 知道向谁报告更新
```

**一句话总结：** 双向关系保证了"自上而下"（渲染流程）和"自下而上"（更新触发）两个方向都能快速找到对方，支持 React 的统一调度和批量更新机制。

### Q3: updateContainer 函数做了什么，有什么作用？

#### 函数作用

**updateContainer**
- **作用**：把要渲染的内容（`element`）包装成更新任务，放入更新队列，触发调度
- **产物**：更新对象（`update`）被加入到 `hostRootFiber.updateQueue`，调度系统启动

#### 代码拆解

```typescript
export function updateContainer(
  element: ReactElementType | null,  // 要渲染的内容（如 <App />）
  root: FiberRootNode                // 应用的根容器
) {
  // 步骤1：找到 hostRootFiber（Fiber 树的入口）
  const hostRootFiber = root.current;
  
  // 步骤2：创建更新对象，把 element 包进去
  const update = createUpdate<ReactElementType | null>(element);
  
  // 步骤3：把更新对象加入到 hostRootFiber 的更新队列
  enqueueUpdate(
    hostRootFiber.updateQueue as UpdateQueue<ReactElementType | null>,
    update
  );
  
  // 步骤4：触发调度，告诉 React "有新任务了，开始干活！"
  scheduleUpdateOnFiber(hostRootFiber);
  
  // 步骤5：返回 element
  return element;
}
```

#### 执行结果

**输入：** 
- `element`：要渲染的内容（如 `<App />`）
- `root`：`FiberRootNode` 对象

**输出：** 返回 `element`（通常用不到）

**执行后产生的效果：**
- `hostRootFiber.updateQueue.shared.pending` → 指向新创建的 `update` 对象
- `update.action` → 包含 `<App />`
- React 的调度系统被触发（`scheduleUpdateOnFiber` 已调用）

#### 数据结构变化

```
执行前：
┌─────────────┐
│FiberRootNode│
│   current   │─→ hostRootFiber
└─────────────┘      │
                     ├─ updateQueue
                     │     └─ shared.pending: null  ← 空队列
                     └─ (其他字段)

执行后：
┌─────────────┐
│FiberRootNode│
│   current   │─→ hostRootFiber
└─────────────┘      │
                     ├─ updateQueue
                     │     └─ shared.pending: update  ← 有任务了！
                     │                        │
                     │                        └─ action: <App />
                     └─ (其他字段)

并且：scheduleUpdateOnFiber(hostRootFiber) 已被调用
```

#### 示例

```typescript
// 假设已经创建了容器
const root = createContainer(document.getElementById('root'));

// 调用 updateContainer
updateContainer(<App />, root);

// 此时：
// 1. hostRootFiber.updateQueue 有了更新任务
// 2. React 开始调度渲染流程
```

#### updateContainer 在整个流程中的位置

```
ReactDOM.createRoot(rootElement).render(<App />)
    ↓
createContainer(rootElement)  ← Q2.1 讲过的
    ↓
updateContainer(<App />, root)  ← 当前这个
    ↓
scheduleUpdateOnFiber(hostRootFiber)  ← 下一步会讲的
    ↓
renderRoot(fiberRootNode)
    ↓
workLoop...
```

**定位：**
- `createContainer` 是"搭建舞台"（创建容器、根节点、队列）
- `updateContainer` 是"发出指令"（创建任务、加入队列、触发调度）

#### 类比理解

**餐厅点餐流程：**

```
你点了宫保鸡丁（<App />）
    ↓
服务员接单（updateContainer）
    ↓
填写点餐单（createUpdate）
    ↓
夹到单子夹上（enqueueUpdate）
    ↓
喊一声"有新单了！"（scheduleUpdateOnFiber）
    ↓
厨师开始做菜（renderRoot）
```

**对应关系：**
- `<App />` = 你点的菜
- `update` = 点餐单
- `updateQueue` = 单子夹
- `scheduleUpdateOnFiber` = 通知厨师

**一句话总结：** updateContainer 就像服务员，把顾客需求（`<App />`）转化成工作单（`update`），放到待办任务里（`updateQueue`），然后通知厨房开工（`scheduleUpdateOnFiber`）。

### Q4: scheduleUpdateOnFiber 做了什么，有什么作用？

#### 函数作用

**scheduleUpdateOnFiber**
- **作用**：接收任意 Fiber 节点，找到应用根节点（FiberRootNode），然后启动渲染流程
- **产物**：触发 `renderRoot`，开始构建 workInProgress 树

#### 代码拆解

```typescript
export function scheduleUpdateOnFiber(fiber: FiberNode) {
  // 步骤1：从当前 Fiber 节点向上找到 FiberRootNode
  const root = markUpdateFromFiberToRoot(fiber);
  
  // 步骤2：启动渲染流程
  renderRoot(root);
}
```

#### 执行结果

**输入：** 
- `fiber`：触发更新的 Fiber 节点（可能是树中任意位置的节点）

**输出：** 无返回值

**执行后产生的效果：**
- 找到了 `FiberRootNode`
- `renderRoot` 被调用，渲染流程启动
- 内部会创建 `workInProgress` 树并开始遍历

#### 内部流程

```
scheduleUpdateOnFiber(任意 fiber 节点)
    ↓
markUpdateFromFiberToRoot(fiber)
    ↓ 向上遍历
找到 hostRootFiber
    ↓ 通过 stateNode
找到 FiberRootNode
    ↓
renderRoot(FiberRootNode)
    ↓
prepareFreshStack(创建 workInProgress)
    ↓
workLoop(开始遍历)
```

#### 示例

```typescript
// 场景1：首次渲染（从 hostRootFiber 触发）
updateContainer(<App />, root);
  ↓ 内部调用

// 场景2：组件更新（从深层组件触发）
function Button() {
  const [count, setCount] = useState(0);
  
  const handleClick = () => {
    setCount(count + 1);  // ← 触发更新
    // 内部会调用 scheduleUpdateOnFiber(buttonFiber)
  };
}
```

#### scheduleUpdateOnFiber 在整个流程中的位置

```
updateContainer(<App />, root)
    ↓
scheduleUpdateOnFiber(hostRootFiber)  ← 当前这个
    ↓
markUpdateFromFiberToRoot(fiber)  ← 下一步会讲的
    ↓
renderRoot(FiberRootNode)
    ↓
prepareFreshStack + workLoop
```

**定位：**
- `updateContainer` 是"发出指令"（创建任务、加入队列）
- `scheduleUpdateOnFiber` 是"启动引擎"（找到根节点、启动渲染）

### Q5: markUpdateFromFiberToRoot 做了什么，有什么作用？

#### 函数作用

**markUpdateFromFiberToRoot**
- **作用**：从任意 Fiber 节点出发，向上遍历到根节点（hostRootFiber），然后通过 `stateNode` 找到 FiberRootNode
- **产物**：返回 `FiberRootNode` 对象（或 `null`）

#### 代码拆解

```typescript
function markUpdateFromFiberToRoot(fiber: FiberNode) {
  // 步骤1：初始化遍历指针
  let node = fiber;
  let parent = node.return;
  
  // 步骤2：向上遍历，直到没有父节点
  while (parent !== null) {
    node = parent;
    parent = node.return;
  }
  
  // 步骤3：检查是否到达 hostRootFiber（tag === HostRoot）
  if (node.tag === HostRoot) {
    // 步骤4：通过 stateNode 返回 FiberRootNode
    return node.stateNode;
  }
  
  // 步骤5：如果不是 HostRoot，返回 null
  return null;
}
```

#### 执行结果

**输入：** 
- `fiber`：任意 Fiber 节点（可能是 Button、div、App 等）

**输出：** 
- 成功：`FiberRootNode` 对象
- 失败：`null`（理论上不会发生）

#### 遍历过程图示

```
场景：Button 组件触发更新

Button Fiber (起点)
    ↑ return
div Fiber
    ↑ return
App Fiber
    ↑ return
hostRootFiber (tag === HostRoot) ← 遍历到这里，parent 为 null
    ↓ stateNode
FiberRootNode ← 返回这个
```

#### 为什么要向上遍历？

在 Q2.4 中我们已经解释过：**React 的更新必须从根节点开始**，因为：
1. 需要检查父组件是否也受影响
2. 调度系统在 FiberRootNode 上
3. 支持批量更新

所以必须先找到 FiberRootNode，才能统一调度。

#### 示例

```typescript
// 假设有这样的组件树：
// FiberRootNode
//   ↓ current
// hostRootFiber (HostRoot)
//   ↓ child
// AppFiber
//   ↓ child
// divFiber
//   ↓ child
// ButtonFiber

// 从 ButtonFiber 开始查找
const root = markUpdateFromFiberToRoot(buttonFiber);

// 遍历路径：
// buttonFiber → divFiber → AppFiber → hostRootFiber
//                                          ↓ stateNode
//                                    FiberRootNode ✅
```

#### markUpdateFromFiberToRoot 在整个流程中的位置

```
scheduleUpdateOnFiber(fiber)
    ↓
markUpdateFromFiberToRoot(fiber)  ← 当前这个
    ↓ 向上遍历
hostRootFiber
    ↓ stateNode
FiberRootNode
    ↓ 传给
renderRoot(FiberRootNode)
```

**定位：**
- `scheduleUpdateOnFiber` 是"启动引擎"
- `markUpdateFromFiberToRoot` 是"找到控制中心"（向上找到 FiberRootNode）

### Q6: createWorkInProgress 做了什么，有什么作用？

#### 函数作用

**createWorkInProgress**
- **作用**：基于 current 树的某个节点，创建或复用对应的 workInProgress 节点（双缓冲机制）
- **产物**：返回 workInProgress FiberNode（新树的节点）

#### 代码拆解

```typescript
export const createWorkInProgress = (
  current: FiberNode,      // current 树的节点
  pendingProps: Props      // 新的 props
): FiberNode => {
  // 步骤1：检查是否已有 alternate（复用）
  let wip = current.alternate;
  
  if (wip === null) {
    // 分支1：首次渲染（mount）- 创建新节点
    wip = new FiberNode(current.tag, pendingProps, current.key);
    wip.type = current.type;
    wip.stateNode = current.stateNode;
    
    // 建立双向连接
    wip.alternate = current;
    current.alternate = wip;
  } else {
    // 分支2：更新（update）- 复用节点
    wip.pendingProps = pendingProps;
    wip.flags = NoFlags;  // 清空副作用标记
  }
  
  // 步骤2：同步其他字段（首次和更新都需要）
  wip.type = current.type;
  wip.updateQueue = current.updateQueue;
  wip.child = current.child;
  wip.memoizedProps = current.memoizedProps;
  wip.memoizedState = current.memoizedState;
  
  return wip;
};
```

#### 补充：为什么要清空 flags？

**原因：避免执行上次渲染遗留的副作用**

当复用节点时（update 分支），这个 `wip` 是**上次渲染用过的**，可能还保留着上次的 flags：

```typescript
// 上次渲染后，wip 可能有：
wip.flags = Placement | Update;  // 需要插入 + 更新 DOM

// 如果不清空：
// → 这次渲染会错误地执行上次的副作用
// → 比如重复插入、重复更新等

// 清空后：
wip.flags = NoFlags;
// → 这次重新标记需要的副作用
// → 每次渲染"从零开始"标记
```

**flags 的生命周期：**

```
第1次渲染（mount）：
1. 创建 wip，flags = NoFlags
2. beginWork：标记 flags（如 Placement）
3. commit：执行副作用（插入 DOM）

第2次渲染（update）：
1. 复用节点（current.alternate）
2. ⚠️ flags 还保留上次的值
3. 清空：wip.flags = NoFlags  ← 必须清空
4. beginWork：重新标记这次的 flags
5. commit：执行副作用（更新 DOM）
```

#### 执行结果

**输入：** 
- `current`：current 树的某个 FiberNode
- `pendingProps`：新的 props

**输出：** 
- workInProgress FiberNode（新树的对应节点）

**首次渲染时（mount）：**
```
current (hostRootFiber)
    ↓ 创建新节点
wip (新的 FiberNode)
    ↓ 建立连接
current.alternate = wip
wip.alternate = current
```

**更新时（update）：**
```
current (hostRootFiber)
    ↓ 复用已有节点
wip (current.alternate)
    ↓ 清空状态
wip.flags = NoFlags
wip.pendingProps = 新 props
```

#### 双缓冲机制图示

```
首次渲染（mount）：
┌─────────────┐
│   current   │
│(hostRootFiber)│
└─────────────┘
      ↓ createWorkInProgress
┌─────────────┐
│     wip     │ ← 新创建
└─────────────┘
      ↕ alternate（建立双向连接）

后续更新（update）：
┌─────────────┐         ┌─────────────┐
│   current   │ ←─────→ │     wip     │
└─────────────┘ alternate└─────────────┘
                           ↑
                      直接复用，不创建新的
```

#### 为什么需要 createWorkInProgress？

**原因：实现双缓冲（Double Buffering）**

- **current 树**：当前显示在屏幕上的树
- **workInProgress 树**：正在构建中的新树

**好处：**
1. 渲染过程可以中断（不影响 current）
2. 渲染完成后一次性切换（更流畅）
3. 节点可以复用（性能更好）

#### 示例

```typescript
// 场景1：首次渲染
const hostRootFiber = root.current;  // current 树的根
const wip = createWorkInProgress(hostRootFiber, {});
// → 创建新的 FiberNode
// → hostRootFiber.alternate = wip
// → wip.alternate = hostRootFiber

// 场景2：第二次渲染（更新）
const wip2 = createWorkInProgress(hostRootFiber, {});
// → 复用 hostRootFiber.alternate（就是上次的 wip）
// → 清空 flags
// → 更新 pendingProps
```

#### createWorkInProgress 在整个流程中的位置

```
renderRoot(FiberRootNode)
    ↓
prepareFreshStack(root)
    ↓
createWorkInProgress(root.current, {})  ← 当前这个
    ↓ 返回
workInProgress (新树的根)
    ↓
workLoop(遍历 workInProgress)
```

**定位：**
- `markUpdateFromFiberToRoot` 是"找到控制中心"
- `createWorkInProgress` 是"复制一份工作台"（创建新树的节点）

### Q7: FiberNode 中的 memoizedState、memoizedProps 等字段有什么作用？

#### 字段分类

FiberNode 中新增的字段可以分为三类：

**1. Props 相关：**
- `pendingProps` - 新的 props（即将生效）
- `memoizedProps` - 旧的 props（已经生效）

**2. State 相关：**
- `memoizedState` - 组件的状态
- `updateQueue` - 更新队列

**3. 双缓冲相关：**
- `alternate` - 指向另一棵树的对应节点
- `flags` - 副作用标记

#### 每个字段的作用

##### 1. pendingProps vs memoizedProps

```typescript
pendingProps: Props;          // 新的 props
memoizedProps: Props | null;  // 旧的 props
```

**作用：用于对比 props 是否变化**

```typescript
// 渲染流程中
function beginWork(wip: FiberNode) {
  const oldProps = wip.memoizedProps;  // 上次的 props
  const newProps = wip.pendingProps;   // 这次的 props
  
  // 对比是否需要更新
  if (oldProps === newProps) {
    // props 没变，可以跳过更新
    return bailout(wip);
  }
  
  // props 变了，需要重新渲染
  return updateComponent(wip);
}
```

**生命周期：**
```
开始渲染：
├─ pendingProps: 新 props（传入的）
└─ memoizedProps: 旧 props（上次的）

完成渲染（completeWork）：
└─ memoizedProps = pendingProps  ← 新 props 变成旧 props
```

##### 2. memoizedState

```typescript
memoizedState: any;
```

**作用：保存组件的状态**

不同类型的组件，`memoizedState` 存储的内容不同：

```typescript
// 函数组件：保存 hooks 链表
function Counter() {
  const [count, setCount] = useState(0);  // ← 存在 memoizedState
  const [name, setName] = useState('');   // ← 存在 memoizedState
}
// memoizedState = { hook1: {state: 0}, hook2: {state: ''} }

// 类组件：保存 state 对象
class Counter {
  state = { count: 0 };  // ← 存在 memoizedState
}
// memoizedState = { count: 0 }

// HostRoot：保存 element（要渲染的内容）
// memoizedState = <App />
```

##### 3. updateQueue

```typescript
updateQueue: unknown;
```

**作用：保存更新队列**

```typescript
// 在 hostRootFiber 上
hostRootFiber.updateQueue = {
  shared: {
    pending: update  // 待处理的更新
  }
}

// update 对象：
update = {
  action: <App />  // 要渲染的内容
}
```

**为什么需要队列？**
- 支持批量更新（多个 setState 合并）
- 支持优先级调度
- 支持并发模式

##### 4. alternate

```typescript
alternate: FiberNode | null;
```

**作用：连接 current 树和 workInProgress 树**

```typescript
// current 树
currentFiber.alternate = wipFiber;

// workInProgress 树
wipFiber.alternate = currentFiber;
```

**用途：**
1. **复用节点**：`createWorkInProgress` 时通过 `alternate` 复用
2. **对比更新**：`beginWork` 时通过 `alternate` 对比新旧节点
3. **树切换**：渲染完成后，`current` 和 `wip` 通过 `alternate` 切换

##### 5. flags

```typescript
flags: Flags;
```

**作用：标记需要执行的副作用（DOM 操作）**

```typescript
// 常见的 flags：
NoFlags = 0b0000000;       // 无副作用
Placement = 0b0000001;     // 插入 DOM
Update = 0b0000010;        // 更新 DOM
Deletion = 0b0000100;      // 删除 DOM

// 使用示例
fiber.flags = Placement;              // 需要插入
fiber.flags = Placement | Update;     // 需要插入 + 更新
```

**生命周期：**
```
beginWork：
└─ 标记 flags（如 Placement）

completeWork：
└─ 可能添加更多 flags

commit 阶段：
└─ 根据 flags 执行 DOM 操作
```

#### 字段之间的关系

```
渲染流程：

1. 准备阶段
   ├─ pendingProps: 新 props 传入
   └─ alternate: 找到对应的 current 节点

2. beginWork 阶段
   ├─ 对比 pendingProps 和 memoizedProps
   ├─ 处理 updateQueue 中的更新
   ├─ 更新 memoizedState
   └─ 标记 flags

3. completeWork 阶段
   ├─ memoizedProps = pendingProps
   └─ 可能添加更多 flags

4. commit 阶段
   └─ 根据 flags 执行副作用
```

#### 示例：完整更新流程

```typescript
// 场景：Button 组件更新

// 1. 触发更新
<Button count={1} />  → <Button count={2} />

// 2. 创建 workInProgress
const wip = createWorkInProgress(currentFiber, { count: 2 });
// wip.pendingProps = { count: 2 }  ← 新 props
// wip.memoizedProps = { count: 1 } ← 旧 props（从 current 复制）
// wip.alternate = currentFiber

// 3. beginWork
if (wip.pendingProps !== wip.memoizedProps) {
  // props 变了，标记需要更新
  wip.flags |= Update;
}

// 4. completeWork
wip.memoizedProps = wip.pendingProps;  // { count: 2 }

// 5. commit
if (wip.flags & Update) {
  // 执行 DOM 更新
  updateDOM(wip);
}
```

#### 一句话总结

这些字段支撑了 React 的**双缓冲机制**和**增量渲染**：
- **Props 字段**：对比新旧 props，决定是否更新
- **State 字段**：保存组件状态和更新队列
- **双缓冲字段**：连接两棵树，标记副作用

### Q8: 为什么要单独配置 hostConfig，有什么作用？

#### 配置回顾

在步骤 2 和步骤 3 中，我们做了两件事：

**步骤 2：创建 hostConfig.ts**
```typescript
export type Container = any;
```

**步骤 3：配置 tsconfig.json**
```json
{
  "compilerOptions": {
    "baseUrl": "./packages",
    "paths": {
      "hostConfig": ["./react-reconciler/src/hostConfig.ts"]
    }
  }
}
```

#### TypeScript paths 配置详解

##### 配置结构

```json
{
  "compilerOptions": {
    "baseUrl": "./packages",           // 基础目录
    "paths": {
      "hostConfig": ["./react-reconciler/src/hostConfig.ts"]
      // ↑ 别名    ↑ 实际路径（相对于 baseUrl）
    }
  }
}
```

##### 解析过程

**代码中使用别名：**
```typescript
import { Container } from 'hostConfig';  // ← 使用别名
```

**TypeScript 解析：**
```
1. 看到 'hostConfig' 不是相对路径（没有 ./ 或 ../）
2. 去 paths 配置中查找 "hostConfig"
3. 找到：["./react-reconciler/src/hostConfig.ts"]
4. 拼接：baseUrl + 路径
   = "./packages" + "./react-reconciler/src/hostConfig.ts"
   = "./packages/react-reconciler/src/hostConfig.ts"
```

**实际效果等同于：**
```typescript
import { Container } from './packages/react-reconciler/src/hostConfig.ts';
```

##### 为什么用数组？

```json
"hostConfig": ["./react-reconciler/src/hostConfig.ts"]
             ↑ 为什么是数组？
```

**支持多个后备路径（fallback）：**
```json
"paths": {
  "hostConfig": [
    "./react-dom/src/hostConfig.ts",      // 优先找这个
    "./react-native/src/hostConfig.ts",   // 找不到找这个
    "./react-reconciler/src/hostConfig.ts" // 都找不到用这个
  ]
}
```

TypeScript 会按顺序尝试，找到第一个存在的文件。

#### 为什么要这样配置？

##### 原因1：跨平台支持（最核心）

**hostConfig 是 React 的"平台适配层"**，不同平台有不同的实现：

```
React Reconciler（核心逻辑）
    ↓ 依赖
hostConfig（平台适配层）
    ↓
├─ react-dom    → 浏览器平台（DOM 操作）
├─ react-native → 移动端平台（Native 组件）
└─ react-art    → Canvas 平台（绘图）
```

**不同平台的 Container 类型不同：**

```typescript
// react-dom 的 hostConfig
export type Container = Element | Document;  // DOM 元素

// react-native 的 hostConfig
export type Container = number;  // Native 组件的 ID

// react-art 的 hostConfig
export type Container = CanvasRenderingContext2D;  // Canvas 上下文
```

##### 原因2：解耦核心逻辑和平台实现

**React Reconciler（核心）不关心具体平台：**

```typescript
// fiber.ts 中导入 hostConfig
import { Container } from 'hostConfig';  // ← 注意：不是相对路径

export class FiberRootNode {
  container: Container;  // ← 使用 hostConfig 的类型
  // ...
}
```

**好处：**
- Reconciler 的代码不需要修改
- 切换平台只需要替换 hostConfig
- 平台实现可以独立开发和测试

##### 原因3：简化导入路径

**对比：**

```typescript
// ❌ 不使用 paths（相对路径）
// fiber.ts 中
import { Container } from './hostConfig';

// fiberReconciler.ts 中
import { Container } from './hostConfig';

// 其他目录的文件
import { Container } from '../../react-reconciler/src/hostConfig';

// ✅ 使用 paths（别名）
// 任何文件中都可以用
import { Container } from 'hostConfig';
// → TypeScript 自动解析到正确路径
```

#### 实际使用场景

**场景1：开发浏览器版 React**

```json
// tsconfig.json
"paths": {
  "hostConfig": ["./react-dom/src/hostConfig.ts"]
}
```

```typescript
// react-dom/src/hostConfig.ts
export type Container = Element | Document;

export function createInstance(type: string) {
  return document.createElement(type);
}

export function appendChild(parent: Container, child: Element) {
  parent.appendChild(child);
}
```

**场景2：开发 React Native**

```json
// tsconfig.json
"paths": {
  "hostConfig": ["./react-native/src/hostConfig.ts"]
}
```

```typescript
// react-native/src/hostConfig.ts
export type Container = number;

export function createInstance(type: string) {
  return NativeModules.UIManager.createView(type);
}

export function appendChild(parent: Container, child: number) {
  NativeModules.UIManager.setChildren(parent, [child]);
}
```

**核心代码（Reconciler）不需要改：**
```typescript
// fiber.ts（两个平台共享这份代码）
import { Container } from 'hostConfig';  // ← 自动找到对应平台的实现

export class FiberRootNode {
  container: Container;  // ← 类型自动适配
}
```

#### 类比理解

**插座标准：**

```
电器（React Reconciler）
    ↓ 需要电源
插座适配器（hostConfig）
    ↓
├─ 中国插座（react-dom）     → 220V 两/三脚
├─ 美国插座（react-native）  → 110V 两扁脚
└─ 欧洲插座（react-art）     → 230V 两圆脚
```

- **电器**不关心插座形状，只需要"插座接口"
- **插座适配器**负责把电器连接到具体的插座
- **更换地区**时，只需要换适配器，电器不变

#### 项目结构示意

```
项目根目录/
├─ packages/                          ← baseUrl
│  ├─ react-reconciler/
│  │  └─ src/
│  │     ├─ hostConfig.ts            ← 目标文件
│  │     ├─ fiber.ts                 
│  │     │   import from 'hostConfig' ← 使用别名
│  │     └─ fiberReconciler.ts       
│  │         import from 'hostConfig' ← 使用别名
│  └─ shared/
└─ tsconfig.json
   {
     "baseUrl": "./packages",
     "paths": {
       "hostConfig": ["./react-reconciler/src/hostConfig.ts"]
     }
   }
```

#### 当前实现

```typescript
// 当前 hostConfig.ts
export type Container = any;
```

**为什么是 `any`？**
- 因为我们还没有实现具体的平台
- 后续实现 `react-dom` 时，会改成 `Element | Document`
- 现在先用 `any` 占位，让代码能跑起来

#### 一句话总结

**hostConfig 是 React 的平台适配层**，通过 TypeScript 的 `paths` 配置实现路径别名，让 Reconciler（核心逻辑）可以跨平台工作，不同平台只需要提供自己的 hostConfig 实现，核心代码不用改。
